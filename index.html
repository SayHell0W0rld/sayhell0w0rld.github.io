用了两个月Claude Code，聊聊它帮我想明白的那些事
工具 2026/6/8

为什么开始用Claude Code

今年四月份开始用Claude Code。原因很直接——手上的项目越来越大，光是理解现有架构就要花不少时间，更别说做改动了。之前试过Cursor，也用过Copilot，但它们的定位更多是”代码补全”或者”在IDE里开个聊天窗”。Claude Code不一样，它跑在终端里，能直接操作文件系统、执行命令、读写代码。简单说，它不只是帮你写代码，它帮你”做项目”。

两个月下来，我对它的使用逐渐从”让它帮我改bug”变成了”让它帮我做架构决策”和”让它帮我管理整个重构流程”。这个转变，跟Claude Code的架构设计有很大关系。

Claude Code的核心架构

Claude Code的架构思路跟很多AI编程工具不同。大多数工具走的是”多agent协作”路线——搞一堆小agent，一个负责写代码、一个负责测试、一个负责review。Claude Code反其道而行，核心就是一个单线程的主循环。

具体来说，它内部跑的是一个while循环：思考 → 调用工具 → 观察结果 → 再思考。循环不断转，直到模型判断任务完成、不再需要调用工具时终止。这个设计看起来简单，但实际用起来非常可控。因为你随时能看到它在做什么，也随时能打断它。

状态管理方面，Claude Code维持的是单一的消息历史记录，不搞复杂的对话分支。这意味着你不用担心它在某个分叉里跑偏了、回不来。用户可以在它执行过程中随时插入新指令——按Escape键打断，它会停下来等你重新指示。这个”随时转向”的能力在实际使用中非常有用，尤其是做架构调整的时候，方向变了直接说一声就行。

上下文构建方式

这是我觉得Claude Code最聪明的设计之一。它不像IDE插件那样预先索引整个代码库（那种方式在大项目里会很慢），而是像人一样，用grep、ls、find这些命令动态探索。第一次打开一个陌生项目时，它会先问你项目在哪，然后开始扫目录结构，读关键文件，逐步建立对项目的理解。

这种方式的好处是：它理解代码的过程是”渐进式”的，跟你自己看项目的方式很像。你告诉它”src/payment/下有个bug”，它会先ls那个目录，读相关文件，然后才开始改。不会一上来就读几百个文件然后说”我准备好了”——那样上下文窗口很快就满了。

工具体系

Claude Code内置了20多个工具，覆盖文件读写、代码编辑、shell命令执行、git操作等。此外支持MCP（Model Context Protocol）扩展外部工具，比如数据库查询、浏览器自动化、Slack消息发送。

但更关键的不是工具数量，而是权限管理。所有写操作和敏感命令都需要用户确认——你设好白名单之后，常用命令可以自动执行，其他的会弹出提示让你确认。这套机制让你敢于把比较大的任务交给它，因为你知道它不会偷偷改你不该改的东西。

上下文压缩

长时间使用时，对话历史会越来越长，逼近token上限。Claude Code有一套4级压缩策略：最便宜的是直接截断旧消息（Snip Compact），成本为零但丢信息；往上是选择性清理（Microcompact）、分层压缩（Context Collapse），最贵的是让AI总结历史（Auto-Compact）。

实际使用中，我大概每30-40条消息会跑一次/compact，手动触发一下。它会把之前的对话总结成摘要，腾出上下文空间。这个设计让长会话不会因为上下文溢出而崩溃。

CC Switch：多模型切换的效率工具

Claude Code本身只支持Claude系列模型（Opus、Sonnet、Haiku），但在国内使用时经常遇到额度用完、网络不稳定等问题。CC Switch是一个开源工具，专门解决多模型切换的痛点。

它的核心功能很简单：在界面上配好多个模型供应商（Claude官方、智谱GLM、Kimi K2、DeepSeek等），点一下就能切换。不需要手动改配置文件，不需要重启终端。

本地代理机制

CC Switch的原理是在本机启动一个本地代理服务。你把Claude Code的API地址指向127.0.0.1:15721（CC Switch的本地端口），所有请求先经过CC Switch转发，再到达你选的模型供应商。切换模型时，CC Switch直接改转发目标，Claude Code这边完全无感。

这种方式的好处是：

  • 配置一次就行，之后都是点一下切换
  • 支持多设备同步（通过坚果云等云盘同步配置目录）
  • WSL环境也能用，Windows和Linux共享同一套配置

实际使用场景

我现在的使用模式是：日常开发用Sonnet（便宜、速度快），做架构设计时切到Opus（推理能力更强），处理简单任务时用国产模型（额度更充裕）。一天切个五六次是常态，如果手动改配置的话确实很烦，CC Switch基本上5秒搞定。

有一个细节值得一提：Claude Code支持用/model命令在会话内切换模型。但这个切换有时候会在Opus额度用完后自动降级到Sonnet，而且不通知你。如果你正在做需要深度推理的任务，降级后质量会明显下降。CC Switch的好处是可以精确控制用哪个模型，不会被自动降级。

架构设计能力：Plan Mode

这是我最看重的功能。Claude Code有一个Plan Mode（规划模式），按两次Shift+Tab进入。进入后，Claude不会直接写代码，而是先分析你的需求，输出一份方案让你审核。方案里会列出它打算改哪些文件、用什么策略、可能的风险点。

这个流程跟人类架构师的工作方式很像：先出方案，确认了再动手。Anthropic的开发者关系负责人说过他90%的时间都在用Plan Mode。

实际操作

举个例子。我有一次需要把一个单体服务拆分成微服务。用Plan Mode，我告诉Claude：”把src/下的服务拆成三个独立模块：auth、payment、notification。每个模块独立部署，共享数据库但各自有自己的表前缀。”

Claude花了大概两分钟分析项目结构，然后输出了一份方案：先识别所有数据库查询，按模块分组；然后创建三个独立的服务目录；最后修改路由和依赖注入。方案里还标出了几个需要手动处理的地方（比如共享的中间件）。

确认方案后，Claude开始执行。整个过程大概跑了40分钟，期间我按了几次Escape做了调整（比如它一开始想把某个utils模块直接复制三份，我让它改成共享库）。最终产出的代码结构清晰，测试全部通过。

跟其他工具的对比

Cursor和Copilot也能做类似的事情，但它们的”规划”更多是在对话框里说说，不会真的帮你拆解任务、逐步执行。Claude Code的不同在于：它是真的在操作你的代码库——读文件、改文件、跑测试、commit。整个过程是”可执行的方案”，不是”建议你这样做”。

项目管理能力：CLAUDE.md和SubAgent

CLAUDE.md：项目的记忆卡

每个项目根目录放一个CLAUDE.md文件，里面写项目的基本信息：目录结构、构建命令、代码规范、常用命令等。Claude Code启动时会自动读取这个文件，相当于给它一份”项目手册”。

用/init命令可以让Claude自动扫描项目并生成CLAUDE.md模板。之后你可以根据实际情况修改。我一般会写这些内容：

  • 项目用的技术栈和版本
  • 目录结构说明
  • 测试命令（npm test、pytest等）
  • 代码规范（ESLint配置、Python的ruff等）
  • 常见的坑和注意事项

这个文件让Claude Code不用每次重新”认识”你的项目。你跟它说”修一下auth模块的bug”，它立刻知道auth模块在src/auth/下，测试用npm test -- --grep auth。

SubAgent：分工协作

SubAgent是Claude Code在7月推出的特性。简单说就是让Claude Code创建独立的子agent，每个子agent有自己的指令、工具权限和上下文。

一个典型的用法是：让一个SubAgent专门做代码审查（只读权限，不能改代码），另一个SubAgent专门写测试。这样主线程可以继续做其他事情，不用等它们完成。

我一般会配置3-4个SubAgent：

  • code-reviewer：代码审查，只读
  • test-writer：写单元测试
  • doc-writer：更新文档

配置方式是在.claude/agents.json里写，每个agent指定描述、模型、指令、权限。用的时候在对话里说”用code-reviewer审查刚才的改动”就行。

Background Tasks

用&后缀可以把任务放到后台执行。比如refactor API routes &会启动一个后台agent重构路由，你可以同时做其他事情。多个后台agent可以在隔离的Git worktree里并行工作，互不干扰。

两个月的整体感受

Claude Code不是那种”装上就飞”的工具。它需要你花时间理解它的工作方式，配置好CLAUDE.md，想清楚什么任务交给它、什么任务自己做。

两个月下来，我的使用习惯是这样的：

  1. 架构设计：用Plan Mode。先描述需求，看Claude的方案，确认后再执行。这是它最值钱的能力。
  2. 日常开发：直接用自然语言描述需求，让它写代码。简单的功能实现非常快，复杂的逻辑需要多轮对话。
  3. 重构：先用CLAUDE.md描述清楚项目结构，然后给Claude一个明确的重构目标。它会自己探索代码、制定计划、逐步执行。
  4. 测试：用SubAgent写测试。让一个专门的agent负责测试，不会干扰主线程的开发。
  5. 模型选择：日常用Sonnet，需要深度推理时切Opus。CC Switch让切换成本几乎为零。

最大的收获是：Claude Code帮我建立了”先规划、后执行”的工作习惯。以前我经常是一边写代码一边想架构，写到一半发现方向错了要重来。现在我会先用Plan Mode让Claude出方案，确认了再动手。这个习惯节省的时间远比Claude Code帮我写代码省的多。

背景：为什么关注 Agent Harness

最近在读 Anthropic 发布的一系列关于 Agent Harness 的工程博文，从 2024 年底的 Building Effective Agents 到 2025 年底的 Effective Harnesses for Long-Running Agents，再到 2026 年初的 Harness Design for Long-Running Application Development。这些文章串在一起看，能看到一条清晰的演进线索：模型能力在飞速进步，但真正决定 Agent 能不能落地做事的，是包裹在模型外面的那套工程系统。

LangChain 在 2026 年 3 月发了一篇 The Anatomy of an Agent Harness，里面引用了一个数据：同一个模型，换一套 Harness 架构，Terminal Bench 2.0 的通过率从 52.8% 拉到 66.5%。模型权重没动一个字节，纯靠外围工程改了跑分。这个数字挺震撼的，说明 Harness 不是什么锦上添花的东西，它是 Agent 能力的核心乘法因子。

下面是我读完这几篇博文后的技术梳理，侧重核心贡献、技术方案和实验结果。

1. Anthropic 的问题定义：长程 Agent 的四重困境

2025 年 5 月，Anthropic 内部想让 Claude 从零搭一个完整的 Web 应用——不是改 Bug，是搭一个产品。任务跑几个小时，单个上下文窗口根本装不下全程。每开一次新 Session，上一轮的记忆就清零。

他们最初按 Context Engineering 的思路来：用外部化记忆（进度文件、功能清单）+ 压缩策略。结果全面溃败，暴露了四种失败模式：

  1. 提前交卷（Premature Completion）：Agent 做了几个功能就宣布”项目完成”，它把已有的代码量当成了完工指标。
  2. 环境盲区（Environmental Blindness）：Agent 在写代码，但环境有 Bug，它写的东西跑不起来，自己不知道。
  3. 虚标完成（False Completion）：功能清单上标了 done，但实际功能是坏的。单元测试过了，端到端根本跑不通。
  4. 失忆实习生（Memory Loss）：每一轮新 Session 都花大量 Token 重新摸索项目结构，像新员工反复问”代码在哪个文件夹”。

这个发现很关键：Context Engineering 解决的是”信息存不住”的问题，但 Agent 的问题远不止存不住。它有时不翻记事本，翻了也不一定照做，做完了也没人验证。记事本不是问题，管理金鱼翻记事本的流程才是问题。

2. 解决方案一：双 Agent Harness（2025 年 11 月博文）

Anthropic 在 Effective Harnesses for Long-Running Agents 中提出了一个双 Agent 架构：

Initializer Agent（初始化 Agent）

只跑一次，做环境搭建：

  • 分析需求，拆出 200+ 个功能点
  • 生成一份结构化的 JSON 功能清单（每条功能有 "passes": false 字段）
  • 初始化 Git 仓库、进度文件 claude-progress.txt、启动脚本 init.sh

为什么用 JSON 而不是 Markdown？Anthropic 发现模型更不容易在 JSON 文件中做不恰当的修改。JSON 的结构化特性天然起到了”防作弊”的作用——Agent 只能改 passes 字段，不能删功能、不能改描述。

Coding Agent（编码 Agent）

每次 Session 只做三件事：

  1. 启动协议：跑 pwd、读 git log、读 progress.txt、读功能清单、执行 init.sh
  2. 增量开发：每次只做一个功能，做完提交 Git，更新进度文件
  3. 留好交接：Session 结束时环境必须处于”可合并”状态——无大 Bug、代码有序、文档清晰

这个设计的核心思路是让单次会话变得足够简单。通过外部状态和任务分解，模型不需要同时记住所有事情，只需要看清单、选一个未完成的功能、做完、交班。

实验效果

改进后 Agent 能连续跑几个小时，逐步完成复杂功能。但仍然有局限：某些功能（比如涉及浏览器原生 alert 弹窗的 Bug）因为工具约束难以检测。

3. 解决方案二：三 Agent Harness（2026 年 3 月博文）

在 Harness Design for Long-Running Application Development 中，Anthropic 把架构进一步推到了三 Agent 模式，灵感来自 GAN（生成对抗网络）的反馈循环：

三个 Agent 的分工

  1. Planner（规划者）：把短 Prompt 扩展成完整的产品 Spec，高层抽象，避免过早陷入技术细节。
  2. Generator（生成者）：按 Sprint 实现 Spec。每个 Sprint 结束时自评，然后交给 QA 验证。用 Git 管版本。
  3. Evaluator（评估者/QA）：用 Playwright MCP 和运行中的应用交互，按 Sprint Contract 测试，按产品深度、功能、设计、代码质量打分。硬性阈值——任何维度不及格就要求 Generator 返工。

关键设计决策

分离生成和评估。Anthropic 发现让 Agent 自己评价自己的工作效果很差——它对自己的产出过于宽容。把生成和评估拆开后，单独调优一个”严格”的 Evaluator 要比让 Generator 自我批判容易得多。

Sprint Contract。每次 Sprint 开始前，Generator 和 Evaluator 就”什么算完成”达成共识，定义可测试的行为。这避免了范围漂移。

Context Reset vs Compaction。对于 Claude Sonnet 4.5，彻底清空上下文、用结构化交接单启动新 Agent（Context Reset），效果优于摘要压缩（Compaction）。但到了 Opus 4.5，模型的长上下文能力更强了，可以用自动压缩做连续 Session。随着模型进步，Harness 的复杂度可以降低。

实验结果：Retro Video Game Maker

方案 耗时 成本 结果
单 Agent 20 分钟 $9 核心功能坏了，UI 差，空间浪费
三 Agent Harness 6 小时 $200 功能完整、UI 精致、集成了 AI 资产生成

Evaluator 提供了具体可操作的 Bug 报告（比如填充工具失效、快捷键逻辑错误、API 路由冲突），直接指导了修复。

4. 训练 Harness vs 生产 Harness：一个被忽视的陷阱

Han Lee 在 Hidden Technical Debt of AI Systems: Agent Harness 中提出了一个重要的区分：

维度 训练 Harness 生产 Harness
目标 探索动作空间，生成优化信号 约束动作空间，确保安全可靠
动作空间 最大化（让模型随便试） 最小化（白名单，默认拒绝）
工具 原始、底层、易扩展 封装、限定范围、有版本
失败处理 欢迎失败（优化信号） 抑制失败（fail closed）
“好”的定义 策略在分布外改进 用户任务成功无事故

关键结论：训练 Harness 应该最宽，生产 Harness 应该最窄，两者之间的差距应该是有意设计、有审计的工程产物，而不是意外。

这个区分解释了为什么有些模型在研究环境中表现很好，部署后却一塌糊涂——他们用了训练时期的宽 Harness 直接上线。

5. Bitter Lesson 在 Harness 上的体现

Lee 还指出，Harness 中的复杂编排逻辑正在被模型的原生能力吸收。历史模式：

  • 2023 年 RAG Pipeline：Harness 管理分块、嵌入、向量库、重排器（因为模型上下文太小）
  • 2024 年 Workflow + Tool Wrapper：Harness 用 Planner-Executor 循环封装 API（因为工具调用不稳定）
  • 2025-2026 年：模型原生就能交错做规划、执行、反思，复杂的封装、Workflow、多 Agent 拓扑变成了开销

Hyung Won Chung 的那句话很到位：**”为当前算力水平添加结构，然后移除它，因为结构会成为下一个算力水平的瓶颈。”**

这解释了为什么 Anthropic 在 Opus 4.6 上可以直接去掉 Sprint 拆分——模型自身的规划和自评能力已经足够强，Harness 不需要再替它做这些事。

6. 实用模式总结

从这几篇博文里，我能提炼出几个反复出现的 Harness 模式：

状态外置（State Pinning）

不要信任模型的记忆。把任务进度、代码变更、测试结果都写到文件系统里（Git、JSON、txt），用版本控制做 checkpoint。文件系统有一种被低估的工程美德：稳定、透明、可恢复。

每次 Session 的启动协议

新 Agent 上线时强制执行：确认目录 → 读历史 → 读进度 → 启动环境 → 才开始干活。像工厂换班时翻交接簿。

生成-评估分离

自评效果差，单独做一个严格的 Evaluator 更好调。Evaluator 可以用 Playwright 等工具直接和运行中的应用交互，做端到端验证。

Success is silent, failures are verbose

Hooks 在 Agent 执行关键操作后运行。类型检查通过了，Agent 听不到任何消息；失败了，错误信息注入上下文触发自我修正。

随模型进步做减法

Harness 的最佳实践不是一成不变的。模型每一代能力提升，Harness 的某些控制组件就该被移除——它们变成了瓶颈而非助力。

7. 对行业趋势的观察

读完这些材料，我有几点感受：

Harness 正在成为产品本身。 模型通过 API 谁都能拿到，但工具链、上下文管理、权限控制、验证器、恢复机制不是谁都能搭好的。LangChain 那个 Terminal Bench 2.0 的数据已经说明了：同一个 Opus，不同 Harness 差出十几个百分点。

Harness 也是技术债务。 Lee 的文章提醒我们，当前 Harness 中的大部分控制逻辑会随着模型进步而过时。如果你把 Harness 当成永久产品表面来做，会花一年时间把它们全拆掉。好的设计是让 Harness 可拆卸、可演进。

核心能力是乘法关系。 任何一项接近零，总能力就接近零。模型能力、上下文管理、工具设计、验证器、恢复机制——它们不是加法关系。这也解释了为什么某些 benchmark 一年内出现断崖式提升：不是模型一夜聪明了几十倍，而是缺失的乘法因子被密集补齐了。

Harness Engineering 不是在替代 Prompt Engineering，而是层层递进。 Prompt Engineering 管”怎么把话说对”，Context Engineering 管”模型开始思考之前，把需要的信息准备好”，Harness Engineering 管”完整的运行时治理”。三层是嵌套关系。

用 Hermes Agent 搭自动化工作流：记忆和技能是怎么工作的

为什么 AI Agent 总是”失忆”

用过 Claude、ChatGPT 的人都知道一个痛点：每次新开一个对话，之前的上下文全没了。上周跟它讲过的技术栈选型、代码规范、部署偏好，它一概不记得。你得重新说一遍，像跟一个每天失忆的同事合作。

这不是模型的问题，模型在单次对话里记忆力很好。问题在于 Agent 框架层面没有设计持久化的记忆机制。大部分 AI 助手还是”无状态”的——对话结束，记忆归零。

Hermes Agent 解决这个问题的方式很直接：把记忆拆成两个文件（MEMORY.md 和 USER.md），持久化到磁盘，每次新会话启动时注入 system prompt。同时用 SQLite + FTS5 建立所有历史会话的全文索引，需要的时候可以搜索过去任何一次对话的内容。加上一个技能系统（Skills），让 Agent 能从经验中提取可复用的操作流程。

下面我详细讲讲这套机制是怎么运转的。

记忆系统：两层文件 + 全文搜索

Hermes Agent 的记忆分三个层级：

第一层：MEMORY.md（Agent 的工作笔记）

这个文件存 Agent 自己的观察和经验，比如：

  • 用户的项目用什么技术栈
  • 这台机器装了什么工具
  • 上次部署踩了什么坑
  • 某个命令在什么环境下会出错

容量限制 2,200 字符（约 800 tokens）。Hermes Agent 通过内置的 memory tool 来管理这个文件，支持 add、replace、remove 三种操作。写入时会做安全扫描，防止 prompt injection 和凭证泄露。

关键设计：memory 是有上限的。当容量超过 80%，Agent 需要主动合并重叠条目或删除过时信息。这不是 bug，是有意为之——有上限迫使 Agent 只保留真正重要的信息，避免记忆膨胀成一堆无用的噪音。

第二层：USER.md（用户画像）

这个文件存关于用户的信息：偏好、沟通风格、工作习惯。容量 1,375 字符。

举个例子，如果我总是在让 Hermes 帮我写代码时要求”简洁回答，不要解释原理”，它会把这条偏好记到 USER.md。下次新会话开始，它就知道该怎么回复我。

第三层：FTS5 全文搜索（历史会话检索）

MEMORY.md 和 USER.md 是”常驻记忆”——每次会话都会加载到 system prompt。但有些信息不需要每次都加载，比如”上周三我们讨论过的那个 Redis 连接池配置”。

Hermes Agent 把所有历史会话存进 SQLite 数据库，用 FTS5 做全文索引。需要回忆某个具体细节时，Agent 可以调用 session_search tool 搜索过去的对话，检索速度大约 20ms。搜索结果由 LLM 做摘要，只返回相关部分。

这三层配合的效果是：

记忆类型 容量 用途 成本
MEMORY.md 2,200 chars 关键环境事实、教训 每次会话固定消耗
USER.md 1,375 chars 用户偏好、风格 每次会话固定消耗
session_search 无限 历史对话回溯 按需调用，不消耗 prompt token

MEMORY.md 和 USER.md 合计约 1,300 tokens，占用 system prompt 的固定开销。这个开销换来的是 Agent 每次都能”记得”关键上下文，不用从零开始。

记忆管理：写入审批机制

Hermes Agent 支持一个 write_approval 配置项。设为 true 时，Agent 写入记忆前需要用户确认。在 CLI 模式下会弹出内联确认；在 Telegram/Discord 等消息平台上，写入会暂存为 “staged” 状态，用户通过 /memory pending 查看待确认的写入，用 /memory approve <id> 批准。

这个机制对团队协作场景很有用——你不希望 Agent 自动把生产环境的密码写进记忆文件。

技能是怎么被创建的

Hermes Agent 不需要你手动写技能。当你完成一个复杂任务、遇到错误后恢复、或者纠正了 Agent 的行为时，Agent 会自动判断这个流程是否值得保存为技能。

触发条件包括：

  1. 完成了复杂任务：比如第一次搭建一个完整的 CI/CD pipeline
  2. 遇到错误并恢复：Agent 从失败中学到了排错步骤
  3. 收到用户纠正：比如”不对，我们的部署方式是 git push 而不是 CLI 命令”
  4. 发现非平凡工作流：某个操作序列不是简单的单步调用

创建的 Skill 会保存到 ~/.hermes/skills/ 目录。下次遇到类似场景，Agent 会自动加载对应的 Skill，按照保存的流程执行。

Progressive Disclosure：按需加载

不是所有技能都同时加载到 prompt 里。Hermes Agent 用三级渐进加载（Progressive Disclosure）来控制 token 消耗：

  • Level 0：只加载技能列表（名称、描述、分类），约 3k tokens
  • Level 1：Agent 判断需要某个技能时，加载完整内容
  • Level 2：加载技能的特定引用文件

这意味着即使你有几十个技能，也不会把 prompt 撑爆。Agent 只在需要时才加载具体技能的内容。

技能自维护：Curator

Hermes Agent 内置一个 Curator，在后台运行，负责：

  • 合并内容重叠的技能
  • 归档长期未使用的技能
  • 不会自动删除任何技能（只归档）

这防止了技能目录随着时间膨胀成一堆冗余文件。

记忆 + 技能 = 自我改进循环

记忆和技能不是孤立工作的，它们构成一个闭环：

举个具体例子

假设我让 Hermes Agent 帮我部署一个 Node.js 项目。

第一次：

  • Agent 不知道项目细节，问了很多问题
  • 找到部署方式是 git push 触发 CI，不是手动部署
  • 部署过程中遇到 node_modules 缓存问题，加了 --no-cache
  • 部署成功

记忆更新：

  • MEMORY.md 写入：”项目用 git push 触发 CI 部署，不是 CLI 命令”
  • 创建 Skill deploy-checklist：包含完整的部署步骤和 --no-cache 的坑

第二次：

  • Agent 自动加载 MEMORY.md，知道用 git push 方式
  • 加载 deploy-checklist skill，按流程执行
  • 直接部署成功，不需要我再解释

这就是自我改进的实际含义。 不是模型被重新训练了，而是 Agent 层面的上下文管理变得更好了。

欧盟CRA法规解读：你的产品需要满足哪些网络安全要求？
技术研究 2025/9/15

最近在整理欧盟网络安全法规时，我把CRA（Cyber Resilience Act）的正式文本过了一遍。这是欧盟针对产品安全领域的重大立法变化，从2024年底正式生效，到2027年底全面执行。写了这篇博文，把核心内容理清楚。

一句话说清楚CRA是什么

CRA的正式编号是Regulation (EU) 2024/2847，它要求所有带数字元素的硬件和软件产品在投放欧盟市场时，必须满足强制性的网络安全要求。产品生命周期全程覆盖——从设计、开发到上市、维护，都要管。

简单说：以前产品安全是”自愿做”，现在是”法律要求你做”。

适用范围：谁需要关注

CRA适用于几乎所有带数字元素的产品，具体包括：

  • 消费电子：智能手机、笔记本电脑、智能电视
  • IoT设备：智能手表、智能门锁、智能摄像头、工业IoT设备
  • 网络设备：路由器、交换机、调制解调器
  • 软件产品：操作系统、应用程序、工业控制软件（独立销售或与硬件捆绑都算）
  • 嵌入式固件：各种联网固件

不适用的情况：已经有专门法规管的领域可以豁免。包括：

  • 汽车（已有UN R155）
  • 航空（已有EASA相关法规）
  • 医疗设备（已有MDR/IVDR）
  • 专为国家安全或军事目的开发的产品

另外一个容易混淆的点：CRA管的是产品安全，NIS2管的是运营者安全。如果你做的是云服务/SaaS，适用的是NIS2而不是CRA。

产品分类：不同风险，不同要求

CRA把产品按网络安全风险等级分成四类：

非关键产品（约90%的产品）：比如硬盘、基本游戏软件、不带安全功能的智能家居设备。只需要满足基本要求，可以自我声明。

Class I 重要产品：包括身份管理软件、浏览器、密码管理器、路由器、操作系统、带安全功能的智能家居设备。这些产品的核心功能对其他产品/网络的安全有影响，需要满足更严格的要求。

Class II 重要产品：包括虚拟化系统（虚拟机监控程序、容器运行时）、防火墙、入侵检测/预防系统、抗篡改微处理器。合规要求最高。

关键产品：包括硬件安全模块、智能电表网关、安全加密处理设备、智能卡/安全元件。这些产品对关键基础设施有直接依赖，需要第三方机构评估。

实际操作中，大部分产品属于非关键产品，走自我声明就行。但如果你做的是路由器、防火墙这类产品，就要注意了。

核心技术要求

CRA附录I列出了具体的技术要求，我挑几个重点的说：

安全设计要求：

  • 攻击面最小化：从设计阶段就开始限制外部接口
  • 默认安全配置：出厂不带默认密码，最小权限原则
  • 身份验证和访问控制：适当的控制机制防止未授权访问

数据保护要求：

  • 静态和传输数据的加密
  • 数据完整性保护：防止未授权篡改
  • 数据最小化：只处理预期用途所需的最少数据

漏洞处理要求：

  • 建立漏洞处理流程
  • 发布安全更新时公开披露已修复漏洞的信息
  • 提供联系地址供用户报告漏洞
  • 及时免费分发安全补丁

文档要求：

  • 技术文档：包含风险评估、安全设计说明等所有相关数据
  • SBOM（软件物料清单）：维护软件组件的详细清单，支持CycloneDX或SPDX格式
  • 符合性声明（DoC）：声明产品符合CRA要求
  • CE标志：通过评估后贴上

合规路径：怎么证明合规

CRA提供了两条合规路径：

自我声明（Self-Assessment）：适用于非关键产品和Class I产品（存在协调标准的情况下）。制造商自己进行评估，起草符合性声明，贴上CE标志。

第三方评估（Third-Party Assessment）：适用于Class II产品和关键产品。需要由欧盟认可的公告机构（Notified Body）进行评估。

不管走哪条路，制造商都需要：

  1. 对产品进行全面的风险评估
  2. 制定并实施安全开发流程
  3. 建立漏洞处理和报告机制
  4. 编写完整的技术文档
  5. 起草符合性声明
  6. 贴上CE标志

报告义务：24小时窗口

CRA引入了一个很严格的报告要求：如果产品中存在被积极利用的漏洞，或者发生影响产品安全性的事件，制造商必须在发现后24小时内向ENISA（欧盟网络安全机构）和国家CSIRT（计算机安全事件响应团队）报告。

这个时间窗口非常短。如果你的产品还没有建立安全事件响应流程，现在就需要开始准备了。

供应链责任

CRA明确要求制造商对供应链安全负责。现代软件开发中，80%-90%的代码库来自第三方组件（包括开源组件），CRA要求：

  • 维护SBOM，明确记录所有软件组件及其版本
  • 对第三方组件进行尽职调查
  • 发现组件漏洞后及时修复并向用户分发更新
  • 定期对产品进行安全测试和审查

这意味着如果你的产品集成了第三方库或SDK，你需要持续跟踪这些组件的安全状态。

罚款和执行

不合规的后果很严重：

  • 所有产品类别：最高1500万欧元或全球年营业额的2.5%（取较高者）；文档不完整等其他违规：最高1000万欧元或全球年营业额的2%；提供误导性信息：最高500万欧元或全球年营业额的1%
  • 监管机构可以下令从市场撤销或限制销售非合规产品
  • 市场监督机构会主动进行合规检查

关键时间节点

  • 2024年12月10日：法案正式生效
  • 2026年6月11日：公告机构通知义务生效
  • 2026年9月11日：报告义务生效（24小时报告漏洞利用事件）
  • 2027年12月11日：所有规定全面执行

现在距离全面执行还有一段时间，但对于需要第三方评估的产品（Class II和关键产品），准备周期可能更长，因为需要预约公告机构。

我的几点观察

与EN 18031的关系：EN 18031是针对RED（无线电设备指令）的协调标准，CRA是上位法，CRA将来会有自己的专用协调标准。这两个标准不能混为一谈。

对中小企业的影响：CRA对中小企业有专门的考虑，但基本要求是一样的。如果你们公司开发的是面向欧盟市场的联网产品，不管规模大小，都需要开始准备。

开源软件的特殊处理：CRA对非商业性开源软件有豁免，但如果开源项目有商业实体在背后运营，情况会复杂一些。

标准还在制定中：CRA的技术要求对应哪些具体标准，目前还在标准化过程中。ENISA和JRC已经发布了CRA要求与现有标准（如EN IEC 62443）的映射文档，可以参考。

总体来说，CRA是欧盟在产品安全领域从”自愿”走向”强制”的标志性立法。如果你的产品带数字元素并且要在欧盟市场销售，现在就应该开始评估合规差距了。

  4. LLM输出要格式化。提示词里明确要求输出JSON，并加代码做格式转换，能避免很多下游节点的报错
  5. 本地调试是必须的。Coze不支持在线调试代码节点，所有代码都要在本地先跑通

DeepSeek-R1 试用体验：当模型开始"思考"
工具 2025/2/10

2025年1月底，DeepSeek-R1发布。我第一时间去试了。

背景

DeepSeek-R1是一个开源的reasoning model，671B参数的MoE架构，每个token只激活37B参数。它在数学、代码和逻辑推理任务上达到了和OpenAI o1相当的水平，而API价格只有o1的3-5%。

开源，MIT协议，免费商用。

这些数字很吸引人，但真正让我感兴趣的是它的”深度思考”模式——模型在回答问题之前，会先把思考过程展示出来。

第一印象：思考链真的很长

打开chat.deepseek.com，选择”深度思考”模式，提了一个稍微复杂的问题。

模型没有像GPT-4那样直接开始输出答案，而是先进入了一个漫长的思考阶段。在<think>标签里，模型写出了它的推理过程——它在分析问题的哪些部分、考虑了哪些可能的方向、遇到了什么矛盾、然后如何修正。

这个过程有时候会持续几万字。

说实话，第一次看到这么长的思考链，我有点不适应。等了十几秒甚至几十秒才看到最终答案。但看完思考过程之后，我发现一个有意思的事：模型会在思考中自我纠正。

比如问它一道数学题，它可能会先尝试一个方法，发现走不通，然后说”这条路不对，换一个思路”，接着尝试另一个方向。这和以前用ChatGPT完全不同——以前的模型是直接给你一个答案，对了就是对了，错了就是错了，你不知道它中间经历了什么。

和普通模型的核心区别

用了一段时间之后，我总结出DeepSeek-R1和普通LLM（比如GPT-4o、Claude 3.5）的主要区别：

普通模型是”快思考”。 你问一个问题，它直接基于训练数据和pattern matching生成回答。有时候它也会输出看似推理的过程，但那个过程本质上只是”答案的一部分”——它不是真的在思考，而是在预测你期望看到的推理步骤。

R1是”慢思考”。 它在生成最终答案之前，真的会花大量token去分析问题、验证假设、排除错误路径。这个过程不是答案的组成部分，而是独立的”思考阶段”。

这一点在DeepSeek的技术报告里有解释：R1的训练过程中，模型通过reinforcement learning自主发现，如果在<think>标签内多停留、多思考，就更有可能获得正确的最终答案（从而拿到更高的reward）。这个”顿悟”不是人为设计的，是模型自己学到的。

实际使用中的感受

几个场景的实际体验：

数学题： 这是R1最擅长的领域。AIME 2024数学竞赛，R1的正确率是79.8%，和o3的96.7%差距不大。它会一步步列出推导过程，遇到矛盾会回头检查。

代码问题： 不错，但不是最强项。对于算法题，R1的思考链很清晰，能展示出它如何分析问题、选择数据结构、考虑边界条件。但对于复杂的工程问题，它的思考过程有时候会跑偏。

普通对话： 没必要用R1。如果只是写个邮件、翻译个句子，R1的长思考链反而是负担——等半天就为了一个简单的回答。

创意写作： 一般。R1的训练目标是推理准确性，不是文学性。写出来的东西逻辑严密但缺少灵气。

思考链的可信度问题

用了一段时间之后，我也产生了一个疑问：模型展示的思考过程，真的代表了它的内部推理机制吗？

上海交大张拳石教授在一次访谈中提出了一个观点：思维链本质上还是模型的端对端输出。模型说”我先考虑了A，再考虑了B”，但它的实际计算过程是否真的如此？我们并不确定。

这在医疗、法律等高风险领域是一个潜在的问题。你不能仅凭模型自己说”我认真思考过了”就信任它的结论。

不过从实用角度看，思考链至少提供了一个debug的窗口。当模型给出错误答案时，你可以回溯它的思考过程，看看它在哪个环节出了问题，然后针对性地调整prompt。这比以前拿到一个黑箱答案要好得多。

成本和可用性

R1的API定价是input $0.55/百万token，output $2.19/百万token。对比o1的$15/百万token（input），便宜了一个数量级。

但性能方面有个问题：R1的首token延迟比较大。根据基调听云的评测，DeepSeek官方的首token时间在3.8秒以上，比火山引擎等第三方服务慢不少。服务器负载高的时候还会报错。

如果你想用R1的推理能力但又不想等太久，可以考虑通过第三方API服务接入，比如火山引擎、阿里云百炼等。

蒸馏版本

DeepSeek还发布了6个蒸馏版本：1.5B、7B、8B、14B、32B、70B，基于Qwen和Llama架构。这些小模型可以在本地运行，适合资源有限的场景。

EN 18031：欧盟无线电设备网络安全标准解读
技术研究 2024/10/28

背景

EN 18031这个标准从2024年8月发布以来，是欧盟无线电设备领域最受关注的网络安全标准之一。趁这个机会整理一下对这个标准的理解。

先说结论：这个标准虽然是RED指令的协调标准，目前只适用于RED Delegated Act (RED-DA)的场景，但它的技术要求和评估方法对整个欧盟网络安全监管体系的影响远不止于此——Cyber Resilience Act (CRA)的技术标准大概率会沿用EN 18031的框架。

标准的来龙去脉

EN 18031是由CEN/CENELEC联合技术委员会JTC 13 WG8开发的，目标很明确：把RED指令Article 3.3(d)(e)(f)的法律条款翻译成可测试、可评估的技术标准。

RED指令本身是2014年的，Article 3.3(d)(e)(f)是后来加的：

  • (d) 网络保护——设备不能损害网络或滥用网络资源
  • (e) 数据与隐私保护——用户个人数据和隐私要得到保障
  • (f) 防欺诈——设备要支持防欺诈功能

2022年的委托法规(2022/30/EU)详细说明了这些要求，原定2024年8月强制执行，后来推迟到了2025年8月1日。EN 18031在2024年8月底发布，2025年1月28日在OJEU上公布，正式获得协调标准地位。

三个部分，对应三种资产

标准分为三个部分，每个部分对应一种资产类型：

EN 18031-1：网络保护
适用于所有能够通过互联网直接或间接通信的无线电设备。核心要求包括访问控制、认证机制、网络通信安全、固件更新等。这一部分的要求是三个标准中最基础的——基本上只要设备联网了，就逃不掉。

EN 18031-2：个人数据保护
适用范围更广一些，除了联网设备外，还包括儿童看护设备、玩具（符合2009/48/EC的）、以及穿戴设备——注意，穿戴设备即使不联网也可能被这个标准管到，因为处理了个人数据。

EN 18031-3：防欺诈
适用于处理货币、货币价值或虚拟货币的联网无线电设备。这一部分对安全机制的要求更高，标准里明确说”单一方案（比如只做数字签名）不足以保护金融资产”。

有个关键概念要理解：安全资产的要求贯穿三个标准，是通用的。而网络资产、隐私资产、金融资产分别对应三个标准的特定部分。

评估流程

整个评估过程可以分为几个阶段：

资产识别

这一步是制造商自己先做的，他们需要识别出产品涉及哪些安全资产、网络资产、隐私资产、金融资产。检测机构会检查这个识别是否完整。实际操作中，很多厂商在这一步就会漏掉一些接口或功能点。

机制评估

针对每个识别出的资产，按照标准中的安全机制要求逐项评估。每个要求都走两步：先判断是否适用（通过决策树），再检查实施是否充分。

文档审查

需要准备的文件包括：资产识别表、技术设计文档、判决理由陈述（就是为什么你认为某个要求不适用或已经满足）。

技术测试

三类评估：

  1. 概念评估：检查文档和理由是否提供了所需证据。比如网络接口通信矩阵文档是否完整。

  2. 功能完整性评估：用网络扫描器验证所有外部接口是否被正确识别和记录。实际操作中，会跑Nmap之类的工具，然后和厂商提供的文档做比对，看看有没有遗漏的端口或服务。

  3. 功能充分性评估：这是最硬核的部分。需要在设备上运行各种测试——模糊测试、渗透测试、协议分析——来验证安全机制是否真的有效。比如检查设备能不能抵御格式错误数据的攻击，认证机制能不能防暴力破解，更新机制是不是安全的。

协调标准的地位和限制

EN 18031获得协调标准地位后，理论上可以推定符合RED的网络安全要求（presumption of conformity），不用经过Notified Body评估。但这里有个重要的限制：

如果产品触发了任何”受限条款”，这个推定就不成立了，必须走Notified Body。主要的受限条款包括：

  • EN 18031-1：允许用户跳过密码设置
  • EN 18031-2：没有实现儿童数据隐私所需的家长控制机制
  • EN 18031-3：金融交易安全措施低于基线要求

实际项目中，不少厂商为了用户体验想跳过密码设置，但这样做就触发了受限条款，反而增加了合规成本。

与CRA的关系

Cyber Resilience Act (EU) 2024/2847预计2027年12月全面生效。CRA的要求和EN 18031有很多重叠的地方，EN 18031的技术框架很可能会成为CRA协调标准的基础。

如果一个产品已经通过了EN 18031的评估，那么很多证据和文档可以直接映射到CRA Annex I的要求上，不需要重复做一遍。所以现在投入EN 18031合规的工作，从长远来看是划算的。

常见问题

几个在检测过程中比较常见的问题：

文档准备不充分：很多厂商只关注技术实现，忽略了决策树的推理过程和理由陈述。但EN 18031的评估很大程度上是基于文档的，文档不完整直接影响评估效率。

模块vs整机：用了预认证的安全模块不代表整机就自动合规。EN 18031评估的是最终产品的整体安全性，模块的安全机制在整机环境下的表现需要重新验证。

更新机制：安全更新（secure update）是高频考察项。标准要求有回滚机制、签名验证、版本管理等，很多IoT设备在这方面做得比较薄弱。

日志记录：安全事件的日志记录要求比很多人想象的要详细，不是简单记一条error log就行，需要覆盖认证事件、配置变更、异常行为等多个维度。

总结

EN 18031是欧盟对无线电设备网络安全的第一次系统性技术规范，它的影响范围会随着CRA的实施进一步扩大。对于制造商来说，越早把这个标准的要求融入产品开发流程，后面的合规成本就越低。

LoRA微调实践：在Google Colab免费GPU上用中文维基百科微调大模型
技术实践 2024/9/12

为什么需要微调

预训练模型（如 LLaMA、Qwen、ChatGLM 等）在海量文本上学习了通用的语言知识，但这些知识是”泛化”的。当你想让模型在特定领域表现更好——比如回答中文百科知识、遵循特定格式、或者适配某个行业场景——预训练模型的表现往往不够用。

全量微调（Full Fine-Tuning）是直接更新模型的所有参数。听起来很直接，但问题在于资源消耗。以 LLaMA-2-7B 为例：

  • FP16 权重：14GB
  • 梯度：14GB
  • AdamW 优化器状态：28GB

仅训练状态就需要 56GB 以上的 GPU 显存。而 Google Colab 免费版提供的是 T4 GPU，显存只有 16GB。更不用说 70B 参数的模型需要 560GB，得上 8 张 A100。

这就催生了参数高效微调（Parameter-Efficient Fine-Tuning, PEFT）的研究方向：能否只训练极少量的参数，就让模型学会新任务？LoRA 就是这个方向上最成功的方案之一。

LoRA 的核心原理

LoRA 的全称是 Low-Rank Adaptation，低秩自适应。它的核心假设是：微调过程中的权重更新矩阵具有很低的”内在秩”（intrinsic rank）。

具体来说，对于预训练模型中的一个权重矩阵 W₀ ∈ R^{d×k}，全量微调会学习一个更新 ΔW ∈ R^{d×k}。LoRA 的做法是将 ΔW 分解为两个低秩矩阵的乘积：

  • LoRA (r=16)：4096 × 16 + 16 × 4096 = 131,072 参数

QLoRA：4-bit 量化 + LoRA

LoRA 减少了可训练参数，但冻结权重仍然需要加载到 GPU 显存中。LLaMA-7B 的冻结权重以 FP16 存放需要 14GB——这已经超出 Colab 免费版的安全范围。

QLoRA（Quantized LoRA）解决了这个问题：将冻结的基础模型量化到 4-bit，然后在其上注入 LoRA adapter 进行 16-bit 精度的微调。

QLoRA 的三个关键技术：

  1. NF4（NormalFloat4）：专为正态分佈权重量化的 4-bit 数据格式，比传统 INT4/FP4 误差低 10-30%
  2. 双重量化：对量化常数再量化，将开销从 0.5 bit/param 降至 0.127 bit/param
  3. 分页优化器：利用 CPU 内存作为缓冲，防止长序列训练时 OOM

本地RAG系统搭建：LangChain + ChromaDB 实战
技术实践 2024/8/20

最近搭了一个本地跑的 RAG 系统，技术栈是 LangChain + ChromaDB。记录一下搭建过程、搜索原理和调参经验，给想自己动手搭的人一个参考。

先说背景

大语言模型（LLM）的训练数据是有截止日期的，也不可能包含你的私有数据。比如我有一批简历，想让模型根据简历内容回答问题——“这个候选人会 Python 吗？””哪几个人有 3 年以上经验？”——直接问 LLM 肯定不行，它没见过这些文件。

RAG（Retrieval-Augmented Generation，检索增强生成）的做法是：先把文档切块、编码成向量存进向量数据库，用户提问时先检索相关内容，再把这些内容拼进 prompt 让 LLM 回答。这样 LLM 就能基于你的私有数据来生成答案了。

整个流程分两个阶段：

  1. 离线索引阶段：加载文档 → 切分成 chunk → 用 embedding 模型编码成向量 → 存入 ChromaDB
  2. 在线查询阶段：用户提问 → query 也编码成向量 → 向量搜索找到最相关的 chunk → 拼成 prompt → 调用 LLM 生成回答

环境搭建

我用的环境：

  • LangChain：负责串联整个 RAG pipeline
  • ChromaDB：本地向量数据库，纯 Python 实现，不需要额外部署服务
  • sentence-transformers：本地 embedding 模型，不需要调 API
  • Ollama：本地跑 LLM（比如 llama3.1:8b），也可以换成任何你习惯的 LLM

如果你用 OpenAI 的 embedding 和 LLM，把 sentence-transformers 换成 openai 就行，ChromaDB 那边 LangChain 有现成的集成。

数据加载与切分

先加载文档。LangChain 支持 PDF、TXT、Markdown 等格式，简历一般是 PDF：

为什么要切分？ 两个原因：

  1. Embedding 模型对输入长度有限制，太长会截断
  2. 检索粒度太粗的话，一大段无关内容会稀释相关信息，LLM 拿到的上下文质量变差

检索：向量搜索是怎么工作的

RAG 的核心在”Retrieval”——怎么从一堆 chunk 里找到和问题最相关的那几条。

向量搜索（Dense Search）

embedding 模型把文本编码成一个低维稠密向量（比如 nomic-embed-text 输出 768 维）。向量之间的距离（通常是 cosine similarity）代表语义相似度。

为什么叫”语义搜索”？因为它的比较不是基于关键词匹配，而是基于含义。比如你搜”编程语言经验”，即使用的是”Python”这个词出现频率不高的 chunk，只要那个 chunk 在语义上和”编程语言”相关，也能被检索到。

Sparse Search（关键词搜索）

传统的关键词搜索（如 BM25）是另一种方式。它基于词频统计，精确匹配关键词。

两者对比：

维度 Dense Search（向量搜索） Sparse Search（关键词搜索）
匹配方式 语义相似度（cosine） 关键词词频（TF-IDF）
处理同义词 好，”编程语言”能匹配”Python” 差，必须精确命中
处理精确术语 一般，可能漏掉专有名词 好，”etcd server.key”精确匹配
计算速度 向量计算，稍慢 倒排索引，很快
适用场景 自然语言问题、模糊查询 精确技术术语、编号搜索

在我的简历筛选场景中，大部分查询是自然语言（”有机器学习经验的人”），用 Dense Search 效果更好。但有些查询是精确术语（”要求 CKA 认证”），Dense Search 可能漏掉。实际使用中可以两种结合。

LangChain 中的检索配置

LangChain 支持几种 search_type：

  • similarity：默认，返回最相似的 k 个 chunk
  • mmr（Maximal Marginal Relevance）：在相关性基础上增加多样性，避免返回重复内容
  • similarity_score_threshold：只返回超过阈值的结果

对于简历筛选这种场景，我用 similarity + 合理的 k 值就够了。

生成回答

检索到相关 chunk 后，把它们拼进 prompt 让 LLM 生成回答：

chain_type="stuff" 是最简单的策略：把检索到的 chunk 直接拼接在一起塞进 prompt。如果检索结果太多，可能会超过 LLM 的上下文长度限制，这时候可以换用 map_reduce 或 refine 等策略。

我实际测试下来，k=3~5 比较合适。太少可能漏掉信息，太多会引入噪声。

调参技巧

搭起来容易，调好需要耐心。我把遇到的问题整理一下：

1. chunk_size

这是影响最大的参数。

  • 太小（<300 字符）：chunk 太碎，丢失上下文，LLM 读到的每条信息都不完整
  • 太大（>2000 字符）：chunk 太粗，相关信息和不相关信息混在一起，检索精度下降
  • 经验值：简历场景下 6001000 字符比较合适。一般文档 8001200 是个不错的起点

2. chunk_overlap

重叠区域让相邻 chunk 的语义连续，避免信息在边界处断裂。

  • 太小（<50）：边界处信息丢失严重
  • 太大（>chunk_size 的 30%）：冗余过多，浪费存储和计算
  • 经验值：chunk_size 的 15%25%，比如 chunk_size=800 时 overlap=150200

3. k 值（检索数量）

  • k=3：适合简单查询，信息量少
  • k=5~8：适合复杂查询，需要多条信息综合回答
  • k 太大：引入噪声，LLM 回答反而变差
  • 我自己的做法：先设 k=5，根据回答质量微调

4. Embedding 模型选择

不同 embedding 模型对检索质量影响很大：

  • nomic-embed-text：开源，本地跑，质量不错，推荐入门
  • bge-large-zh-v1.5：中文场景效果更好
  • text-embedding-3-small（OpenAI）：质量高，但需要 API key 和费用
  • sentence-transformers/all-MiniLM-L6-v2：轻量，适合英文场景

简历筛选涉及中文，我用的是 bge-large-zh-v1.5，效果比 nomic-embed-text 好一些。

5. LLM 的 temperature

temperature=0 让回答确定性更强，适合需要精确引用的场景。temperature 越高回答越”灵活”，但可能编造不存在的信息。RAG 场景一般设 0 或 0.1。
