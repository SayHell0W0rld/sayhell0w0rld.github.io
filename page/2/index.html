Microsoft 365 Copilot 三个月体验：各应用模块功能实测
工具 2024/6/12

公司上个月升级了 Microsoft 365 高级版本，附带了 Copilot 功能。我用了大概三个月，这篇记录下每个模块的实际表现和遇到的问题。

先说 Copilot 是什么

Microsoft 365 Copilot 不是单独的工具，而是嵌入到 Word、Excel、PowerPoint、Outlook、Teams 这些你每天都在用的应用里的 AI 助手。它背后调用的是大语言模型，能理解自然语言，帮你做一些生成、总结、分析的工作。

关键点是：它是按用户订阅的，每人每月 30 美元。需要有 Microsoft 365 E3/E5 或同等授权才能使用。

各模块实测

Outlook：邮件总结和回复草稿

这是我目前用得最多的场景。遇到那种抄送了十几个人、来回几十封的邮件线程，直接让 Copilot 总结要点和 action items，省了不少时间。

回复邮件也能用。它会根据上下文草拟一封回复，基本的商务用语和格式都没问题。如果内容涉及具体数据或者需要引用内部文件，生成的草稿还是要自己过一遍。

好用的地方：长邮件线程总结、快速回复草稿、跨时区时差场景下的邮件处理。
需要注意：不要让它代替你做最终判断，特别是涉及审批或承诺的邮件。

Teams：会议记录和摘要

Teams 里的 Copilot 目前只支持在开启转录（transcription）的会议中使用。这点很重要——如果会议没有录下来，Copilot 是没法事后分析的。

在会议进行中，你可以直接问 Copilot “到目前为止讨论了哪些决策点”，它会从转录内容里提取出来。会后也能用，比如让它总结会议的 action items，或者把某段讨论整理成文字发到频道里。

好用的地方：自动提取 action items、生成会议纪要、回答”刚才说了什么”。
需要注意：需要提前开启转录功能。如果不想留下转录记录（比如讨论敏感内容），可以用”无转录模式”——Copilot 会在会议结束后删除转录。

Word：文档生成和改写

Word 里的 Copilot 可以根据你提供的素材生成文档初稿。比如你有一份数据报告，让它写一份 500 字的摘要，它会从文档里提取关键信息组织成文。

改写功能比较实用。选中一段文字，Copilot 可以帮你换个语气（正式/轻松）、调整长度、或者换一种表达方式。

好用的地方：基于已有素材生成初稿、文本改写、格式化排版建议。
限制：它没法帮你遵守企业品牌的排版规范。比如你说”用这个模板的样式”，它大概率会无视。公司内部的专有名词拼写规则也管不了。

Excel：数据分析和可视化

Excel 版的 Copilot 目前还在 preview 阶段。基础的数据操作表现不错——比如”帮我找出销售额最高的产品”或者”高亮显示前 10 名”，它能直接操作单元格并解释做了什么。

数据洞察功能也还行，输入”insights for 这个字段”，它会在聊天窗口里给出一个文字分析，可以选择”Add all insights to grid”把结果放到新 sheet 里。

好用的地方：基础筛选排序、快速数据概览、非技术人员的入门数据分析。
限制：复杂的财务建模、估值计算、需要多层公式联动的统计分析，它做不了。如果你是财务或者数据分析岗位，它目前帮不上大忙。

PowerPoint：从文档生成演示文稿

PowerPoint 的 Copilot 可以根据 Word 文档生成演示文稿。它会把内容拆分成多张 slide，配上标题和要点。

好用的地方：快速从文档生成演示文稿框架、内容重组。
限制：只能从 Word 文档（.docx）生成，没法直接用 Excel 或其他格式的素材。生成的视觉效果比较基础，需要自己调整。另外，过渡动画和排版这些细节，它不会自动处理。

跨应用体验的问题

用下来最大的感受是：每个应用里的 Copilot 体验是割裂的。

微软不同团队开发不同应用的 Copilot，导致提示词的逻辑、数据访问能力、功能成熟度都不一样。比如 Word 的 Copilot 能引用你环境里的其他文件，但 Loop 里的 Copilot 连文件都访问不了。

Plugin 功能目前也只在 M365 Chat 里能用，没法从 Word 或 PowerPoint 里调用。这跟最初宣传的能力有差距。

还有一个问题：更新很快但没有 release notes。你可能某天突然发现 PowerPoint 里多了个新功能，但官方没有任何文档说明。这给内部培训和用户预期管理带来了不少麻烦。

一些实际的 Prompt 建议

好的提示词是 Copilot 发挥效果的关键。几个比较实用的写法：

总结类：总结这封邮件的要点，列出所有需要我处理的事项。

生成类：根据这份销售数据，写一份季度业绩摘要，重点强调营收增长和主要驱动因素。

分析类：分析这个表格，找出销售额排名前 5 的产品，用柱状图呈现。

会议类：总结这次会议中所有已做出的决策和分配的行动项目。

核心原则：具体、明确、给上下文。模糊的指令通常得到模糊的结果。

结论

Copilot 不是银弹，不能指望它完全替代人工。但在特定场景下——特别是 Teams 会议总结和 Outlook 邮件处理——它确实节省了时间。

目前各模块的成熟度差异很大，Teams 和 Outlook 最成熟，Excel 和 PowerPoint 还在追赶。对于企业来说，重要的是找到真正能产生价值的用例，而不是强行推广给所有人。

我的建议是：先从 Teams 会议总结和 Outlook 邮件处理这两个场景入手，让团队成员自己体验，然后再逐步扩展到 Word 和 Excel。

OpenDroneID协议研究：无人机远程识别的开放标准
技术研究 2024/5/22

背景

全球无人机数量在过去几年快速增长，从消费级航拍无人机到工业级物流无人机，低空空域的使用密度显著提升。随之而来的问题是：当空中出现一架无人机时，地面人员如何知道它是谁的、从哪里起飞、在执行什么任务？

传统做法是依靠机场塔台的雷达和目视观察，但这些手段对小型消费级无人机基本无效——它们体积太小，雷达反射截面很小，而且很多飞行活动发生在雷达覆盖不到的区域。监管机构需要一种更直接的方式来识别这些”无名”的飞行器。

2021年，美国联邦航空管理局（FAA）正式发布了Remote ID最终规则，要求在美国注册的无人机必须具备广播身份信息的能力。欧盟航空安全局（EASA）也推进了类似的Direct Remote ID法规。日本国土交通省（MLIT）在2022年6月20日起强制要求广播式Remote ID。

这些法规催生了一个关键需求：需要一个开放、可互操作的协议来统一Remote ID的实现方式。OpenDroneID就是这个需求的产物——它是ASTM F3411和ASD-STAN prEN 4709-002标准的开源实现。

什么是Remote ID

Remote ID（远程识别）本质上是给无人机加了一个”电子车牌”。它通过无线电广播的方式，将无人机的身份信息、位置信息、操作者信息等数据发送给周围的接收设备。

Remote ID有两种主要实现方式：

  1. 广播式Remote ID（Direct Remote ID）：无人机直接通过蓝牙或Wi-Fi向周围广播识别信息，不依赖互联网连接。这是OpenDroneID主要覆盖的场景。

  2. 网络式Remote ID（Network Remote ID）：无人机通过蜂窝网络或互联网将识别信息发送到在线服务，接收者通过网络查询获取信息。

FAA的规则主要要求广播式Remote ID。操作员有两种合规路径：购买内置Remote ID功能的新无人机，或者为现有无人机加装独立的Remote ID广播模块。

OpenDroneID协议架构

OpenDroneID定义了6种消息类型，外加1种用于将多条消息打包的Message Pack类型：

1. Basic ID（基本标识）

包含无人机的核心身份信息：

  • UAS ID：无人机的唯一序列号，遵循ANSI/CTA-2063-A标准
  • UA Type：无人机类型，取值包括：
    • Fixed wing（固定翼）
    • Multi-rotor（多旋翼）
    • Helicopter（直升机）
    • Hybrid Lift（混合升力）
    • Other（其他）
  • ID Type：标识符类型（序列号、CAA注册号、UTM执行标识符等）

2. Location（位置）

包含无人机的实时动态位置信息：

  • Latitude：纬度，单位为10^-7度
  • Longitude：经度，单位为10^-7度
  • Altitude Type：高度参考类型
    • Above Takeoff Point（起飞点以上，即AGL）
    • Above Sea Level（海拔高度，即WGS-84椭球高度）
  • Altitude：高度值，单位为米
  • Heading：航向角，单位为度（0-360）
  • Speed：水平速度，单位为m/s
  • VSpeed：垂直速度，单位为m/s（正值上升，负值下降）
  • Timestamp：时间戳，自UTC 2022年1月1日起的秒数

3. Authentication（认证）

用于身份验证和防篡改。协议定义了4种认证页面类型：

  • Account ID（账户ID）
  • Authentication Code（认证码，基于位置/时间的散列）
  • Session ID（会话ID）
  • Message Set Signature（消息集签名）

认证数据在16个页面之间循环传输。

4. Self-ID（自标识）

操作者可选填的明文描述信息，比如”航拍摄影”、”农业喷洒”等用途说明。

5. System（系统）

包含操作者和运营系统的信息：

  • Operator Location Type：操作者位置类型（固定/动态/未知）
  • Operator Latitude/Longitude：操作者位置坐标
  • Operator Altitude：操作者海拔高度
  • Area Count：同时操控的无人机数量
  • Area Radius：运营区域半径
  • Area Ceiling：运营区域高度上限
  • Area Floor：运营区域高度下限
  • Classification：EASA分类（Open/Specific/Certified）
  • Timestamp：时间戳

6. Operator ID（操作者ID）

操作者的注册标识符，如FAA的Remote ID编号。

7. Message Pack（消息包）

将上述多种消息打包为一个整体。Wi-Fi NaN、Wi-Fi Beacon和蓝牙5.0 Extended Advertising使用Message Pack来传输数据，而蓝牙4.x Legacy Advertising由于25字节的广播数据限制，只能逐条发送单个消息。

广播方式

OpenDroneID支持四种物理层广播方式：

蓝牙4.x Legacy Advertising

最基础的广播方式，兼容性最好。但蓝牙4.x的广播数据包限制为31字节，一条Remote ID消息通常超过这个限制，因此需要将数据拆分到多个广播包中发送。接收端需要重组这些包才能还原完整消息。至少每333ms广播一次，以满足每秒3条消息的最低要求。

蓝牙5.0 Long Range with Extended Advertising

蓝牙5.0扩展了广播数据包的长度限制，支持发送完整的Message Pack。同时支持长距离模式（Coded PHY），可以显著扩大广播覆盖范围。欧盟法规将此作为强制要求。

Wi-Fi Neighbor Awareness Networking (NaN)

Wi-Fi NaN（也叫Wi-Fi Aware）是一种设备间直接发现和通信的协议，不需要接入点。无人机通过NaN服务广播Remote ID数据，范围内的接收设备可以自动发现并接收这些数据。支持2.4GHz和5GHz频段。美国和欧盟法规都将此列为强制支持。

Wi-Fi Beacon

在标准Wi-Fi Beacon帧中嵌入Remote ID数据（使用厂商自定义元素）。这种方式不需要设备之间建立连接，任何监听Wi-Fi Beacon的设备都能接收到数据。

与其他标准的关系

OpenDroneID不是一个孤立的标准，它处于多个标准体系的交汇点：

  • ASTM F3411：美国的Remote ID标准，OpenDroneID的v3版本基于ASTM F3411 v1.1（F3411-22a）
  • ASD-STAN prEN 4709-002：欧洲的Direct Remote ID标准，与ASTM F3411 v1.1保持对齐
  • IETF DRIP：IETF的Drone Remote ID Protocol工作组定义了Remote ID的安全和信任框架。已发布的RFC包括RFC 9153（需求和术语）、RFC 9374（RID实现）、RFC 9434（架构）。DRIP在ASTM的广播基础上增加了HIT（Hash-based ID）等安全机制
  • MAVLink：OpenDroneID定义了对应的MAVLink消息，飞控系统可以通过MAVLink将位置和身份信息发送给Remote ID广播模块

各区域法规差异

不同地区对Remote ID的要求存在差异：

特性 FAA规则 ASTM v1.1 EU规则 日本规则
序列号(ANSI/CTA-2063-A) 必需 必需 必需 必需
动态位置 必需 必需 必需 必需
WGS-84高度 必需 必需 - 必需
AGL/起飞点高度 - 可选 必需 可选

安全问题

OpenDroneID的广播数据既没有加密也没有签名验证，这是一个已知的设计权衡。这意味着：

  1. 隐私问题：任何人都能接收和解码无人机的广播数据，包括操作者位置
  2. 伪造攻击：攻击者可以注入伪造的Remote ID数据，制造虚假的无人机存在
  3. 数据篡改：中间人可以修改广播中的位置或身份信息

IETF DRIP工作组正在通过HIT和签名机制解决部分安全问题，但这些方案在当前的OpenDroneID实现中尚未广泛部署。Nozomi Networks等安全厂商已经开发了检测异常ODID流量（如不合理的遥测数据、重复消息、异常流量激增）的能力。

开源实现

opendroneid-core-c是OpenDroneID的官方C语言库，提供了消息编码和解码功能。主要的硬件实现包括：

  • ArduRemoteID：基于ESP32-S3/C3的固件，支持BT4、BT5和Wi-Fi，与ArduPilot飞控集成
  • ESP32实现：基础ESP32仅支持BT4 Legacy Advertising，ESP32-S3/C3支持BT5
  • Linux发送器：支持Wi-Fi NaN的Linux实现

接收端方面，Android有官方的OpenDroneID OSM应用和Dronetag的DroneScanner；iOS方面由于苹果的限制，仅支持BT4 Legacy Advertising的接收。

总结

OpenDroneID作为Remote ID的开源实现，解决了一个实际问题：在无人机数量快速增长的背景下，如何让监管机构和公众识别空中的飞行器。协议本身的技术设计相对简洁——基于蓝牙和Wi-Fi的广播机制，消息格式清晰，实现门槛不高。

但安全方面的缺陷是显而易见的。缺乏加密和认证意味着协议更容易被伪造和攻击，IETF DRIP的扩展方案是否能有效解决这些问题还有待观察。在法规强制推动和实际安全需求之间，OpenDroneID目前选择了先实现基本功能，安全增强留待后续版本。

工作中用到了密码学：TLS 加密套件抓包分析实战
工作 2024/4/8

背景

说来也巧，之前一直觉得密码学是那种离工作很远的东西——什么 RSA 算法、AES 加密，听上去都是安全团队的事。直到最近项目需要排查一个 HTTPS 连接的问题，我才真正动手去抓包分析 TLS 协议的握手过程，发现这套东西其实没那么神秘，而且和日常开发关系还挺大。

为什么要去抓 TLS 的包

起因是我们的服务在对接一个第三方 API 时偶尔会出现连接超时，排查下来发现不是网络层的问题，而是 TLS 握手阶段出了状况。对方服务器支持的加密套件列表和我们客户端的偏好不太匹配，导致协商过程变长甚至失败。

这让我意识到：光知道”用 HTTPS 就安全了”是不够的，得搞清楚 TLS 握手到底在交换什么、选了什么加密套件、证书链是否正确验证。

TLS 握手过程回顾

TLS 握手的核心就是客户端和服务端在建立加密通道之前的一系列”对话”。拿 TLS 1.2 举例，流程大概是这样的：

  1. Client Hello — 客户端发一个消息过去，带着自己支持的 TLS 版本号、随机数、以及一份加密套件列表
  2. Server Hello — 服务端从列表里选一个它也支持的套件，返回随机数
  3. Certificate — 服务端把自己的证书链发过来，客户端拿去验证
  4. Server Key Exchange — 如果用的不是 RSA 直接密钥交换（比如 ECDHE），服务端会发临时公钥参数
  5. Server Hello Done — 服务端说”我说完了”
  6. Client Key Exchange — 客户端生成 pre-master secret，用服务端公钥加密发过去（或者发 DH 参数）
  7. Change Cipher Spec — 客户端通知”接下来我要用新密钥加密了”
  8. Finished — 客户端发一条加密消息，里面包含之前所有握手消息的校验值

服务端收到后也走一遍 Change Cipher Spec + Finished，握手完成，后面就是加密的应用数据了。

加密套件到底是个啥

加密套件的命名其实很规范，比如 TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256，拆开来看：

  • ECDHE — 密钥交换算法（椭圆曲线 Diffie-Hellman 临时密钥）
  • RSA — 签名算法（用来验证身份的）
  • AES_128_GCM — 对称加密算法和模式
  • SHA256 — 哈希算法

TLS 1.3 简化了这个命名，因为密钥交换和签名算法基本固定了（ECDHE + 灵活选择签名算法），套件名只保留加密和哈希部分，比如 TLS_AES_128_GCM_SHA256。

在排查问题时，我用 openssl 查了下双方支持的套件列表：

发现对方服务器默认排序里把一些老套件排在前面，而我们客户端出于安全考虑禁用了那些套件，双方交集变小，握手就慢了。

Wireshark 抓包实战

实际操作中，我用 tcpdump 先抓包：

然后把 pcap 文件拖进 Wireshark 里分析。Wireshark 对 TLS 协议的解析做得很好，展开每一条消息能直接看到：

  • Client Hello 里的 Cipher Suites 列表（一长串套件 ID）
  • Server Hello 里选中的套件
  • Certificate 里的证书链（从叶子证书到根 CA）
  • Server Key Exchange 里的 ECDHE 参数（公钥、曲线类型等）

有意思的一点是 TLS 1.3 有个”中间盒兼容模式”（middlebox compatibility mode）。因为很多网络中间设备（防火墙、代理）只认 TLS 1.2 的格式，TLS 1.3 的 Client Hello 会故意伪装成 TLS 1.2 的 Record 格式，骗过中间设备。抓包的时候能明显看到一条 Change Cipher Spec 消息出现在不该出现的位置，那就是兼容模式在起作用。

几个实际遇到的问题

1. 证书链不完整

第三方服务器返回的证书链缺少中间 CA 证书，导致客户端本地没有对应的中间 CA 根证书去验证。解决办法是让对方在服务器配置里补全证书链，或者在客户端信任该中间 CA。

2. 协议版本降级

对方服务器默认支持 TLS 1.0 和 1.1，但我们客户端最低只接受 TLS 1.2。协商时对方会尝试降级，如果两边都不让步就握手失败。最后我们协商后让对方把 TLS 1.0/1.1 关掉了。

3. SNI 扩展缺失

有次发现抓包里 Client Hello 没带 Server Name Indication 扩展，导致服务端不知道该返回哪个虚拟主机的证书，返回了默认证书，客户端验证不通过就断了。原因是用的 openssl 命令行工具没指定 -servername 参数。

一些心得

通过这次实践，我对密码学在工程中的角色有了更具体的认识。之前觉得加密就是调个库的事，现在知道了：

  • 加密套件的选择直接影响兼容性和安全性，不能随便配
  • 证书验证是个完整的链路，中间任何一个环节断了都会出问题
  • TLS 握手过程中的每个字段都有其作用，抓包分析是最直观的学习方式
  • TLS 1.3 相比 1.2 确实在握手效率和安全性上都有提升，值得在新系统中优先采用

密码学不是只存在于教科书里的理论，它就藏在我们每天用的 HTTPS 连接里。下次遇到网络连接问题，不妨抓个包看看 TLS 握手过程，说不定问题就出在加密套件协商这一步。

重新捡起Python爬虫
技术实践 2024/2/14

有段时间没写爬虫了，最近想重新捡起来。起因是想爬一下QQ空间的历史说说，做一些数据分析的练习。翻了翻以前的代码，发现很多库的用法已经变了，干脆从头梳理一遍。

先说框架选型

Python爬虫这块，主流的工具就那么几个：requests + BeautifulSoup、Scrapy、Selenium。三者定位不同，选哪个取决于你要爬什么。

requests + BeautifulSoup

最轻量的组合。requests负责发HTTP请求，BeautifulSoup负责解析HTML。适合爬静态页面，或者你能拿到API接口的场景。

几行代码就能跑起来，调试也方便。我的经验是，大多数爬虫需求，先用requests试试，很多时候页面直接给你返回HTML，根本不需要渲染JS。

Scrapy

Scrapy是个完整的爬虫框架，自带调度器、中间件、管道这些东西。优点是快，异步架构处理大量页面效率很高。缺点是项目结构比较重，写个简单的爬虫反而不如直接用requests方便。

我之前爬过一个小说网站，用Scrapy写了完整的spider，包括去重、限速、数据导出CSV，跑起来确实舒服。但如果是像QQ空间这种需要登录、有复杂反爬机制的，Scrapy本身的登录处理不太方便，通常还是要配合Selenium用。

Selenium / Playwright

这类工具的核心能力是操控真实浏览器。页面是JS动态渲染的、需要登录的、有验证码的——这些场景用Selenium或Playwright比较合适。

缺点也明显：慢。我测试过同一个页面，requests抓取大概3秒，Selenium要40多秒。因为它要启动一个完整的浏览器进程，等JS执行完再拿DOM。

我的选择

日常爬虫：requests + BeautifulSoup，够用就行。
大规模批量采集：Scrapy，性能好。
需要登录/JS渲染：Selenium或Playwright。

实际上很多时候是组合使用的。比如我这次爬QQ空间，就是Selenium登录拿cookie，然后用requests去请求数据接口。

实战：QQ空间爬虫

QQ空间是个挺好的练习目标，因为它基本集齐了爬虫会遇到的所有难题。

第一步：登录

QQ空间的登录页面嵌在一个iframe里，而且有滑块验证。纯requests是搞不定的，直接上Selenium。

这里有个坑：登录成功后页面会跳转，但Selenium不会自动等待。我一开始没加sleep，直接去取cookie，结果取到的是登录前的空cookie。加了显式等待之后才稳定下来。

第二步：获取g_tk

QQ空间的API请求需要一个叫g_tk的参数，它是从cookie里的p_skey值经过一个哈希算法算出来的。

这个算法本身不难，难的是你知道它存在。我一开始根本不知道有这个参数，请求一直返回403，debug了挺久。后来搜了相关资料才发现，QQ空间的请求必须带上g_tk。

除了g_tk，还需要从页面源码里提取qzonetoken。这个token藏在一段内联JS代码里，用正则可以拿到。

第三步：请求数据接口

拿到cookie、g_tk和qzonetoken之后，就可以用requests去请求QQ空间的数据API了。

返回的是JSONP格式的JSON数据，需要去掉回调函数的包装才能解析。

第四步：处理反爬

QQ空间的反爬主要体现在几个方面：

登录验证：滑块验证，Selenium能处理但需要手动过一次，之后cookie就保存了。

请求频率限制：请求太快会被封。我一开始没加延迟，跑了没多久就被限制了。后来加了随机的sleep，1到3秒随机间隔。

动态加载：QQ空间的内容是通过AJAX接口加载的，不是直接写在HTML里的。所以解析HTML是拿不到内容的，必须找到API接口。

JSONP包装：返回的数据被包在回调函数里，需要截取。我的处理方式是取掉前10个字符和后3个字符（去掉callback(和);）。

第五步：数据清洗

留言板的数据是HTML格式的，里面混着各种标签、script、style。需要写一个清洗函数把HTML标签去掉。

遇到的问题

总结几个值得注意的地方：

  1. Cookie过期：QQ空间的cookie有效期不长，跑一会儿就要重新登录。我后来写了个检测机制，发现请求失败就自动重新登录。

  2. iframe切换：Selenium操作QQ空间必须先切换到iframe里，不然找不到登录元素。操作完再切回来。

  3. 编码问题：中文内容经常出现乱码，requests返回的encoding有时候不对，手动设置response.encoding = "utf-8"能解决大部分问题。

  4. 页面结构变了：QQ空间的HTML结构时不时会变，之前写的选择器过一段时间就失效了。所以解析逻辑最好写得宽泛一点，别依赖太细的层级结构。

  5. robots.txt：虽然QQ空间的robots.txt没有明确禁止爬取，但还是要控制频率，不要给别人服务器造成负担。

小结

重新写了一遍爬虫，发现和几年前比，基本原理没变，但工具和生态变化不少。requests依然好用，Selenium从2.x升级到了4.x，Playwright成了新的选择，Scrapy也出了很多插件。

对于QQ空间这种有复杂反爬的站点，单靠一个库是搞不定的。Selenium解决登录，requests解决请求效率，正则+JSON解析解决数据提取，三者配合才能跑通。

下次打算试试Playwright替代Selenium，据说在等待策略和性能上都比Selenium好一些。

Attention Is All You Need 论文精读：我理解Transformer架构的全过程
学习笔记 2023/6/18

写在前面

最近在学 Transformer 架构，绕不开这篇 2017 年 Google 团队发表的论文——Attention Is All You Need。说实话，第一次读这篇论文的时候，很多概念似懂非懂，尤其是 Multi-Head Attention 和 Positional Encoding 部分。后来翻了几篇解读博客，加上自己动手写了点伪代码，才算真正理清了整篇论文的脉络。这篇文章就是把我理解的过程记录下来，希望能帮到同样在啃这篇论文的人。

一、为什么需要 Transformer？

RNN 的瓶颈

在 Transformer 出现之前，NLP 领域的主流方案是 RNN（循环神经网络）及其变体 LSTM、GRU。RNN 处理序列的方式是”一个词一个词地看”：每处理一个新词，就把之前所有词的信息压缩进一个固定长度的 hidden state。这种串行处理方式带来了两个根本性的问题：

  1. 无法并行：第 t 个词的计算必须等第 t-1 个词算完，训练速度极慢，多 GPU 也没用。
  2. 长距离依赖：句子越长，早期词汇的信息在 hidden state 里衰减越严重。比如一句话里”它”指代的名词在十个词之前，RNN 很难把这种关联记住。

CNN 在序列任务上也有尝试，但卷积核只能捕捉局部上下文，要处理长距离依赖就得堆很多层，效率同样不高。

Transformer 的核心思路

Transformer 的想法很直接：既然 RNN 的问题是串行，CNN 的问题是局部，那就干脆不用 RNN 也不用 CNN，完全用 Attention 机制来建模序列中的依赖关系。Attention 的好处是，任意两个位置之间可以直接计算关联度，不需要经过中间步骤，天然支持并行。

二、Attention 机制详解

2.1 Scaled Dot-Product Attention

Attention 的核心公式：

这里涉及三个概念：

  • **Q (Query)**：当前词”想要找什么信息”
  • **K (Key)**：每个词”对外展示的标识”
  • **V (Value)**：每个词”实际携带的信息”

计算过程分三步：

  1. 用 Q 和所有 K 做点积，得到一个”相关性分数”向量
  2. 除以 sqrt(d_k) 进行缩放（原因后面说），再做 softmax 归一化成权重
  3. 用权重对所有 V 做加权求和，得到输出

为什么要除以 sqrt(d_k)？ 假设 Q 和 K 的每个元素都是均值为 0、方差为 1 的随机变量，那么 Q 和 K 的点积的均值为 0，方差为 d_k。当 d_k 很大（比如 512）时，点积值可能很大，导致 softmax 输出接近 one-hot 分布——几乎所有权重都集中在某一个位置上，梯度接近 0，模型没法学。除以 sqrt(d_k) 把方差拉回 1，softmax 的输入落在合理范围内。

2.2 Self-Attention（自注意力）

Self-Attention 就是 Attention 机制的一种特殊应用：Q、K、V 全部来自同一个输入序列。

举个例子，处理句子 “The cat sat on the mat because it was tired”，当模型要理解 “it” 这个词时，Self-Attention 会让 “it” 同时去查看句子中所有其他词的 K，发现 “cat” 的 K 和自己的 Q 最匹配，于是从 “cat” 的 V 中提取信息。这样 “it” 的输出向量就融合了 “cat” 的语义。

2.3 Multi-Head Attention（多头注意力）

一个 Attention 头只能学到一种关联模式。但语言中的依赖关系是多样的：语法关系、语义关系、指代关系……需要多个”角度”来捕捉。

Multi-Head Attention 的做法：

  1. 把 Q、K、V 分别用不同的权重矩阵投影到 d_model/h 维（h 是头数，论文中 h=8）
  2. 每个头独立做一次 Scaled Dot-Product Attention
  3. 把 h 个头的输出拼接起来，再用一个线性变换映射回 d_model 维

三、Transformer 整体架构

Transformer 遵循经典的 Encoder-Decoder 框架，原论文用于机器翻译任务：Encoder 把源语言句子编码成上下文向量，Decoder 基于这些向量和已生成的目标语言词汇，逐词生成翻译结果。

3.1 Encoder

Encoder 由 N=6 个相同的层堆叠而成，每层包含两个子层：

  1. Multi-Head Self-Attention：捕捉输入序列内部的全局依赖
  2. Position-wise Feed-Forward Network：对每个位置独立做两层全连接网络

每个子层都带有 残差连接（Residual Connection） 和 层归一化（Layer Normalization）：

残差连接让梯度能直接跨层传播，缓解深层网络的退化问题。

3.2 Decoder

Decoder 同样由 N=6 个层堆叠，每层包含三个子层：

  1. Masked Multi-Head Self-Attention：防止位置 i 关注到位置 i 之后的词（因为生成时还看不到未来的词）
  2. Multi-Head Cross-Attention：Q 来自 Decoder 的上一层，K 和 V 来自 Encoder 的输出——这是 Decoder “查看”输入句子的地方
  3. Position-wise Feed-Forward Network

同样每个子层都有残差连接 + 层归一化。

3.3 Positional Encoding（位置编码）

Self-Attention 本身对输入顺序是”无感”的——你把句子中的词重新排列，每个位置的输出只是跟着重排，不会有任何变化。但语言是有顺序的，”狗咬人”和”人咬狗”意思完全不同。

四、为什么 Self-Attention 比 RNN 好？

论文中给出了一个对比表，核心指标：

指标 RNN CNN Self-Attention
每层复杂度 O(n·d²) O(k·n·d²) O(n²·d)
最短路径长度 O(n) O(log_k(n)) O(1)
可并行化程度 不可 部分 完全
  • 最短路径长度：Self-Attention 中任意两个位置的路径长度是 1，直接就能看到对方；RNN 需要经过 n 步才能从第一个词到达最后一个词。
  • 并行化：Self-Attention 的所有位置可以同时计算，训练效率远高于 RNN。

当然，Self-Attention 的复杂度是 O(n²·d)，序列长度 n 很大时计算量会暴增。这也是后续很多工作（比如 Linformer、Performer）试图解决的问题。

五、训练细节

论文中几个值得注意的训练技巧：

学习率调度

前 warmup_steps 步学习率线性增大（warmup），之后按 step 的 -0.5 次方衰减。warmup 的作用是避免训练初期梯度不稳定时学习率过大导致模型震荡。

标签平滑（Label Smoothing）

训练时不是让模型输出 one-hot 分布（比如 [0, 0, 1, 0]），而是让它输出一个”软”分布（比如 [0.05, 0.05, 0.85, 0.05]）。这能提高模型的泛化能力，防止过拟合。

正则化

论文使用了 Residual Dropout 和 Attention Dropout，dropout rate 为 0.1。

六、实验结果

在 WMT 2014 英德翻译任务上，Transformer (big) 模型取得了 28.4 BLEU，比之前的 SOTA 模型高出 2.0 BLEU，而且训练成本只有前者的 1/40。

在英法翻译任务上，达到了 41.0 BLEU，同样大幅超越之前的所有单模型。

更值得注意的是在 English constituency parsing（句法分析）任务上的表现——这个任务要求模型理解句子的层级结构，和翻译任务差异很大。Transformer 在这个任务上同样表现优异，说明它学到的是语言的通用规律，而非翻译任务的专属模式。

七、写在最后

读完这篇论文，最大的感受是”简单但深刻”。整个架构的核心就是 Attention 一个机制，但通过精心设计（多头、残差、归一化、位置编码），让它在实际效果上碾压了之前的复杂架构。这也解释了为什么这篇论文的影响力如此之大——它不只是解决了一个问题，而是提供了一个通用的序列建模框架，后来的 BERT、GPT、LLaMA 等大模型全部建立在这个基础之上。

图片隐写：用Matlab实现LSB嵌入、提取与抗干扰测试
技术实践 2023/5/22

为什么搞这个

信息安全课的作业需要做图片隐写，要求用Matlab实现。我之前对隐写的理解停留在”改最后一位像素值”这个层面，实际做下来发现事情没那么简单——尤其是抗干扰这块，坑不少。

LSB隐写的基本原理

LSB（Least Significant Bit）是图片隐写里最基础的方法。核心思路：图片每个像素由RGB三个通道组成，每个通道0-255，二进制表示8位。把秘密信息的每一位替换到像素的最低有效位上，人眼看不出区别。

一张512x512的RGB图片有512x512x3 = 786432个LSB位，理论上能藏98304字节（约96KB）的数据。实际因为要放头部信息（比如信息长度），能用的空间更少一些。

为什么LSB能工作？人眼对颜色的变化有感知阈值，RGB值差1（比如从200变到201）根本看不出来。但如果改动太多位（比如改到第4位），肉眼就能察觉颜色偏差了。

Matlab实现：嵌入端

嵌入流程分三步：

  1. 把秘密信息转成二进制串
  2. 遍历图片像素，逐位替换LSB
  3. 保存含密图片

几个注意点：

  • 用de2bi的left-msb参数确保高位在前，提取时顺序才对
  • 长度头用32位，虽然浪费但实现简单，不用纠结终止符的问题
  • bitand(pixel_val, 254)把最低位清零，+ all_bits(i)写入新值，比XOR写法更直观

Matlab实现：提取端

提取是嵌入的逆操作：

提取时有个坑：如果嵌入时没用长度头，提取端就不知道该读多少位。我一开始用0xFFFFFFFF做结束符，结果发现秘密信息里如果包含这个序列就会提前截断。后来改成32位长度头，问题解决。

抗干扰测试

隐写术要实用，光嵌入和提取还不够——图片经过常见操作后，秘密信息还能恢复吗？这才是实际应用中的关键问题。

我测了几种常见的干扰场景：

JPEG压缩

JPEG是有损压缩，会对像素值做DCT变换再量化，LSB位几乎必然被改写。测试结果：经过JPEG压缩（质量因子50）后，LSB隐写的信息几乎全军覆没。

这是LSB方法的固有弱点，因为它把信息放在最容易被改动的位上。要对抗JPEG压缩，需要换到频域隐写（比如DCT域），把信息嵌入到DCT系数中能量较大的位置。

高斯噪声

加了标准差为5的高斯噪声，大约30%的LSB位被翻转。提取出的信息乱码了。

不过如果加了纠错码（比如汉明码或RS码），可以恢复一部分。我在消息后面加了汉明码冗余，噪声标准差5以内能恢复90%以上。

裁剪

裁掉图片右下角10%，提取就完全失败了——因为像素的物理位置变了，第32+1个像素已经不是嵌入的第一个消息位了。

解决办法：不在空间域直接嵌入，改用伪随机序列打乱像素顺序（PN序列隐写）。嵌入时用密钥生成随机序列决定像素访问顺序，提取时用同样密钥重现序列。这样即使裁剪掉一部分，只要剩余部分足够多，就能恢复信息。

旋转

旋转5度后提取完全失败。旋转会导致像素重采样，LSB位被彻底打乱。

这个需要先做旋转校正（图像配准），把图片转回原角度再提取。Matlab里用imregister可以做，但增加了系统复杂度。

滤波

均值滤波（3x3核）后，约60%的LSB位被改变。中值滤波更狠，几乎100%翻转。这是因为中值滤波直接替换了像素值。

提升鲁棒性的思路

从测试结果看，纯LSB隐写的抗干扰能力很弱。要提升鲁棒性，有几个方向：

频域隐写（DCT/DWT）：把信息嵌入到频域系数中。DCT域的中频系数对压缩和滤波有一定鲁棒性，是JPEG隐写的常用方案。DWT（小波变换）域的低频系数对几何变换更稳定。

冗余编码：用纠错码（汉明码、RS码、BCH码）增加冗余。牺牲嵌入容量换取纠错能力。这是频域隐写之外最直接的增强手段。

重复嵌入：同一段信息嵌入多个位置，提取时取多数投票。简单但有效，容量代价大。

自同步机制：加入同步头或特征标记，让提取端能定位嵌入起点。对抗裁剪和旋转时很关键。

遇到的问题

  1. 图片格式：PNG是无损的，用PNG做载体没问题。JPEG本身就有损，LSB信息会被压缩破坏。所以载体图片一定要用PNG或BMP。

  2. 灰度vs彩色：灰度图只有一个通道，嵌入容量只有彩色图的1/3。我一开始用灰度图测试，结果消息稍微长一点就放不下，换成RGB图后好了很多。

  3. 数值溢出：如果像素值是255，最低位是1，嵌入0时要减1没问题。但如果用加法写入，255+1会溢出变成0。所以写LSB时一定要先清最低位再写，不要直接加。

  4. 嵌入顺序：按行优先还是列优先都行，但嵌入和提取必须用同样的顺序。我之前嵌入时用了列优先，提取时用了行优先，提取出来的全是乱码。

总结

LSB隐写实现简单，嵌入容量大，视觉不可感知——这些优点让它成为入门学习的首选。但抗干扰能力弱是硬伤，JPEG压缩、噪声、裁剪、旋转都能轻易破坏信息。

实际应用中，DCT域和DWT域的隐写方案更实用，配合纠错码能达到不错的鲁棒性。LSB更多是作为基线方法，用来理解隐写的基本原理和局限性。

停车场管理系统：一个没有硬件的"智能"停车方案
项目 2023/4/10

做这个毕设之前，我研究了大量停车场系统的论文和开源项目。发现一个普遍现象：大部分停车场管理系统的毕设都是同一个套路——SpringBoot + Vue + 百度API，界面长得差不多，功能也差不多。我不太想做一个”标准答案”式的毕设，就一直在想能不能做点不一样的东西。

选题和调研

停车场管理是个经典毕设题目，功能模块清晰：车位管理、车辆进出、收费计算、数据统计。技术栈也没什么争议，SpringBoot + Vue + MySQL 基本是标配。

但”智能”这两个字怎么体现？大部分方案是接入硬件设备：摄像头拍车牌、地感线圈检测车辆、道闸电机控制抬杆、LED屏显示剩余车位。

我花了两周时间调研这些硬件方案。地磁线圈需要在地面开槽埋线，施工成本高；道闸电机要买成品设备，一个少说几百块；摄像头选型、供电、网络布线也是一堆事情。最关键的是，本科毕设没有场地、没有预算、没有施工条件，这些东西根本落不了地。

调研完硬件方案，我做了一个决定：全部用软件模拟，把”停车场”搬到网页上。

方案设计：纯软件模拟

核心思路是把停车场的物理流程用Web界面重新演绎：

车牌识别：没有摄像头，改成在网页上手动上传一张包含车牌的图片。后端拿到图片后，调用一个开源项目的识别API来返回车牌号。这个开源项目是GitHub上一个基于深度学习的车牌识别项目，提供了HTTP接口，我只需要传图片过去、拿结果回来。识别准确率在清晰图片上还行，模糊的就不行了，但毕设够用。

道闸抬杆：真实场景里车辆驶入时道闸会抬起、车辆通过后落下。我用两张图片（道闸关闭状态和抬起状态）按时间切换来模拟这个过程。前端用定时器每隔几百毫秒切换一次图片，视觉上就是一个道闸抬起的动画效果。简单粗暴，但确实能看出来在”抬杆”。

LED显示屏：道闸旁边通常有个LED屏显示车牌号和剩余车位。我直接在道闸图片上方叠加了一个 <div>，定位到LED屏的位置，把识别出的车牌号填进去。样式上做了点模仿LED字体的效果，绿色字体、黑色背景，远看还有那么点意思。

车辆进出记录：用户上传图片 → 识别车牌 → 如果该车牌没有在场记录则视为入场，创建记录；如果已有在场记录则视为出场，计算停车时长和费用。这个逻辑和真实停车场是一样的，只是触发方式从”摄像头自动识别”变成了”手动上传”。

技术实现

后端用 SpringBoot，主要处理三件事：

  1. 图片接收和转发。接收前端上传的车牌图片，转发给识别API，拿到结果后写入数据库。
  2. 停车记录管理。入场时创建记录，出场时查询记录、计算费用、标记出场。这里用 Redis 做了一层缓存，在场车辆信息放 Redis 里，避免每次都查 MySQL。虽然毕设的并发量几乎为零，但这个设计在面试时可以展开聊。
  3. 车位管理。管理员可以设置总车位数，系统根据在场车辆数动态计算剩余车位。

数据库用 MySQL，主要三张表：用户表、车辆信息表、停车记录表。停车记录表是核心，记录车牌号、入场时间、出场时间、费用、状态等字段。建表的时候纠结了一下出场时间要不要允许 NULL（入场时还没有出场时间），后来加了出场时间可为空的约束。

前端用 Vue + Element UI。管理后台就是标准的后台管理模板：左侧菜单、顶部导航、中间内容区。车牌识别页面做了一个上传区域，用户拖入或选择图片后，页面会显示上传的图片，几秒后在旁边显示识别结果和模拟的道闸动画。

Redis 还承担了另一个角色：存储图片上传后的临时路径。图片先存到服务器本地目录，路径写入 Redis，识别完成后从 Redis 取出路径做后续处理。这样避免了频繁操作文件系统。

遇到的问题

图片上传和预览：Element UI 的 el-upload 组件本身支持图片预览，但我需要在识别完成后再展示结果区域，需要控制组件的状态。调了好几次 on-success 回调的参数才搞对。

道闸动画的时序：道闸抬起、保持、落下这三个阶段的时间间隔需要配合。太快了看不出来，太慢了显得拖沓。最后调整成：识别成功后0.5秒抬杆、保持2秒、0.5秒落下，总共3秒完成一次”过车”。

识别API的错误处理：开源项目的识别API偶尔会返回空结果或者格式异常的JSON。最开始没做好异常处理，前端直接报错白屏。后来在后端加了一层封装，识别失败时返回一个友好的错误信息，前端也加了 loading 状态和错误提示。

Redis 缓存一致性：入场时把车牌号写入 Redis，出场时从 Redis 读取并删除。但如果服务器重启，Redis 里缓存的在场车辆就丢了。毕设答辩时老师问了这个问题，我的回答是”生产环境会用 Redis 持久化或者从数据库恢复”，老师点了点头没追问。

答辩情况

答辩时演示了完整流程：打开网页 → 上传一张车牌图片 → 等待识别 → 看到车牌号显示在LED屏位置 → 道闸图片切换模拟抬杆 → 车辆进入（记录写入数据库）。出场流程反过来操作一遍。

老师的评价是”有点创意”。这个评价比我预期的好。我本来担心纯软件模拟会显得太简陋，但老师认可了”在没有硬件条件下用软件模拟物理流程”这个思路。

不过说实话，这个方案的局限性也很明显：不能处理实际场景中的复杂情况（遮挡、模糊、多角度），道闸动画只是视觉模拟没有实际控制逻辑，LED显示也只是个装饰性的 div。但作为本科毕设，在有限条件下做出了一个能跑通的系统，我覺得还算合格。

一些想法

做完这个毕设最大的感受是：毕设的核心不是技术多牛，而是在有限条件下解决问题的能力。硬件方案做不了，那就换软件方案；摄像头没有，那就用上传替代；道闸控制不了，那就用图片动画模拟。每个”替代方案”都不是最优解，但组合在一起构成了一个完整的演示系统。

如果以后有时间和条件，我可能会把这个系统升级成真正的硬件方案：接一个USB摄像头做实时识别，用Arduino控制道闸电机，加一个树莓派跑本地推理。但那就是另一个项目了。

Stable Diffusion 体验：从零到能写提示词
技术实践 2022/12/15

前言

2022年8月，Stable Diffusion发布当天我就装上了。那时候还没有现在这么多教程和插件，官方release里就一个Python脚本和一行命令。我跑的第一张图是一只猫，用了默认提示词，出来的效果一般——毛发糊成一团，背景乱七八糟。但那个瞬间很明确：这东西能用。

后来几个月我花了不少时间研究提示词怎么写，怎么调，踩了不少坑。这篇记录一下当时的理解和实践经验。

Stable Diffusion的基本工作方式

简单说：你写一段文字描述（prompt），模型把这段文字编码成语义向量，然后用这个向量引导扩散过程（diffusion process）一步步去噪，最终生成一张图片。

所以提示词的质量直接决定了出图质量。模型不理解你的意图，它只看你写了什么词，然后在训练数据里找最接近的分布去生成。

提示词的基本结构

一个好的提示词通常由这几类关键词组成：

  1. 主体（Subject）：你想画什么。越具体越好。”a sorceress”和”Emma Watson as a powerful mysterious sorceress, casting lightning magic, detailed clothing”出来的结果完全不同。
  2. 画风（Medium）：用什么媒介。digital painting、oil painting、photography，这些对最终风格影响很大。
  3. 风格（Style）：艺术流派。hyperrealistic、fantasy、art nouveau等等。
  4. 画家/平台（Artist/Website）：引用特定画家或平台的视觉风格。比如”by Alphonse Mucha”、”trending on artstation”。
  5. 分辨率/细节（Resolution）：highly detailed、sharp focus、8k。
  6. 色调和光照（Color/Lighting）：iridescent gold、cinematic lighting、rim lighting。

一个完整的提示词可能是这样的：

不需要每类都填。画家、网站和风格之间有重叠，根据需要组合就行。

提示词顺序很重要

写提示词的时候，顺序代表优先级。靠前的词权重大于靠后的词。所以最重要的内容要放在前面。

比如你想画一个戴帽子的女孩，”1girl, hat”比”hat, 1girl”的效果更稳定——因为”1girl”锚定了主体，”hat”作为附加元素叠加。

模型的标准token限制是75个。超过75的部分可能被截断。AUTOMATIC1111的WebUI会自动把提示词分成75-token一组处理，但如果一个概念（比如”blue hair”）被切到两个组里，效果就会出问题。可以用BREAK关键词手动分割。

负面提示词：告诉模型不要画什么

负面提示词（Negative Prompt）的作用是引导采样过程远离特定特征。不是把元素从画面里移除，而是在无条件采样阶段让模型往相反的方向走。

四个主要用途：

  1. 移除不想要的物体：比如脸上有胡子，负面提示词加mustache。想更狠一点就加权重：(mustache:1.3)。
  2. 微调已生成的图片：想减少飘动的头发，负面提示词加windy。
  3. 关键词切换：采样中途切换负面提示词内容，比如the: (ear:1.9): 0.5。
  4. 改变整体风格：负面提示词加blur让画面更清晰，加painting或cartoon让画面更写实。

通用的负面提示词模板：

权重控制：微调每个词的影响

提示词里每个关键词都可以调整权重。

WebUI语法：

  • (keyword:1.5) — 加强keyword的效果，1.5倍
  • (keyword:0.5) — 减弱keyword的效果，0.5倍
  • (keyword) — 等价于 (keyword:1.1)，加强10%
  • [keyword] — 等价于 (keyword:0.9)，减弱10%
  • ((keyword)) — 1.1 × 1.1 = 1.21倍

权重的范围一般是0.1到100，但实际用到超过2倍就很少见了。太高的权重容易导致画面崩坏。

注意：WebUI和NovelAI的语法不同。 WebUI用()是加强（1.1倍），NovelAI用{}是加强（1.05倍）。不要混用。

提示词混合：在同一张图里融合两个概念

有一种比较高级的技巧：在采样过程中途切换关键词。

语法：[keyword1 : keyword2 : factor]

比如 [apple : fire : 0.5] — 前一半步数生成苹果的特征，后一半步数生成火焰的特征。最终出来的图是两者的混合体。

这里有个关键点：第一个关键词决定整体构图。第二个关键词只影响细节。所以如果想保持某种构图，第一个词要写清楚。

这个技巧也可以用来融合人脸。比如：

多个名人名字加权重混合，生成一张具有混合特征的脸。

提示词矩阵和调度

Prompt Matrix（提示词矩阵）： 用|分隔关键词，WebUI会自动生成所有关键词组合的图片。比如：

会生成4张图：单独street、street+illustration、street+cinematic lighting、street+illustration+cinematic lighting。适合快速测试不同关键词的效果。

Prompt Schedule（步数调度）： 在采样过程中动态改变提示词。

语法：[from : to : when]

  • [fantasy : cyberpunk : 16] — 第16步之前用fantasy，之后切换为cyberpunk
  • [to : when] — 到when步之后才加入to
  • [from :: when] — 到when步之后移除from

我遇到的问题

  1. 风格冲突：有些标签本身自带固定风格。比如loli标签有特定的画风，加flatcolor会冲突。解决办法是降低冲突标签的权重，或者直接换掉。

  2. 训练数据的隐含关联：blue eyes经常生成白人面孔，因为训练数据里蓝眼睛和白人的关联很强。brown eyes则能生成更多种族多样性。这个在写提示词的时候要注意。

  3. 模型差异：换了checkpoint模型之后，之前好用的风格关键词可能完全失效。每个模型的训练数据不同，对同一个词的理解也不同。换了模型一定要重新测试关键词效果。

  4. 过多关键词反而不好：不是写得越多越好。堆太多词会稀释每个词的效果，模型会随机从一堆词里选。与其写20个词，不如精选5-6个最准确的词。

关于Danbooru标签体系

SD的早期版本（尤其是基于NovelAI训练的模型）大量使用Danbooru的标签体系。Danbooru是一个动画图片托管网站，有标准化的标签系统。这些标签被直接用于模型训练，所以用Danbooru标签来描述画面效果往往很精确。

比如：

  • 1girl — 一个女孩
  • solo — 画面只有一个人
  • long hair, blue eyes — 长发、蓝眼
  • looking at viewer — 看向观众
  • outdoors, beach — 室外、海滩

这套标签体系的好处是标准化程度高，模型对每个标签的理解比较一致。缺点是需要专门学习标签词汇，不太适合用自然语言描述。

后来AUTOMATIC1111有了Tag Autocomplete插件，输入时会自动补全标签。还有人做了中文翻译版的标签库，降低了上手门槛。

一些实用工具

  • Danbooru标签超市 (tags.novelai.dev)：标签浏览和组合工具，支持标签权重调整和混合组配置。
  • Tag Autocomplete插件：在WebUI里自动补全标签，支持中文翻译。
  • MagicPrompt：基于GPT-2训练的提示词补全工具，输入几个关键词就能扩展成完整提示词。

最后

写提示词这件事没有标准答案。模型在变，工具在变，写法也在变。核心思路是：明确你想要什么，用模型能理解的词汇去描述它，然后通过权重和负面提示词去微调。

我现在回头看早期写的那些提示词，很多都很粗糙。但那时候没有教程，就是一次次试错，看效果，调参数，再试。这个过程本身挺有意思的。
