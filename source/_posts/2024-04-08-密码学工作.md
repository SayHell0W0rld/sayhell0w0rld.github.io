## 背景

说来也巧，之前一直觉得密码学是那种离工作很远的东西——什么 RSA 算法、AES 加密，听上去都是安全团队的事。直到最近项目需要排查一个 HTTPS 连接的问题，我才真正动手去抓包分析 TLS 协议的握手过程，发现这套东西其实没那么神秘，而且和日常开发关系还挺大。

## 为什么要去抓 TLS 的包

起因是我们的服务在对接一个第三方 API 时偶尔会出现连接超时，排查下来发现不是网络层的问题，而是 TLS 握手阶段出了状况。对方服务器支持的加密套件列表和我们客户端的偏好不太匹配，导致协商过程变长甚至失败。

这让我意识到：光知道"用 HTTPS 就安全了"是不够的，得搞清楚 TLS 握手到底在交换什么、选了什么加密套件、证书链是否正确验证。

## TLS 握手过程回顾

TLS 握手的核心就是客户端和服务端在建立加密通道之前的一系列"对话"。拿 TLS 1.2 举例，流程大概是这样的：

1. **Client Hello** — 客户端发一个消息过去，带着自己支持的 TLS 版本号、随机数、以及一份加密套件列表
2. **Server Hello** — 服务端从列表里选一个它也支持的套件，返回随机数
3. **Certificate** — 服务端把自己的证书链发过来，客户端拿去验证
4. **Server Key Exchange** — 如果用的不是 RSA 直接密钥交换（比如 ECDHE），服务端会发临时公钥参数
5. **Server Hello Done** — 服务端说"我说完了"
6. **Client Key Exchange** — 客户端生成 pre-master secret，用服务端公钥加密发过去（或者发 DH 参数）
7. **Change Cipher Spec** — 客户端通知"接下来我要用新密钥加密了"
8. **Finished** — 客户端发一条加密消息，里面包含之前所有握手消息的校验值

服务端收到后也走一遍 Change Cipher Spec + Finished，握手完成，后面就是加密的应用数据了。

## 加密套件到底是个啥

加密套件的命名其实很规范，比如 TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256，拆开来看：

- **ECDHE** — 密钥交换算法（椭圆曲线 Diffie-Hellman 临时密钥）
- **RSA** — 签名算法（用来验证身份的）
- **AES_128_GCM** — 对称加密算法和模式
- **SHA256** — 哈希算法

TLS 1.3 简化了这个命名，因为密钥交换和签名算法基本固定了（ECDHE + 灵活选择签名算法），套件名只保留加密和哈希部分，比如 TLS_AES_128_GCM_SHA256。

在排查问题时，我用 openssl 查了下双方支持的套件列表：

```bash
# 查看服务端支持的加密套件
openssl s_client -connect example.com:443 -cipher 'ECDHE+AESGCM'
```

发现对方服务器默认排序里把一些老套件排在前面，而我们客户端出于安全考虑禁用了那些套件，双方交集变小，握手就慢了。

## Wireshark 抓包实战

实际操作中，我用 tcpdump 先抓包：

```bash
sudo tcpdump -i eth0 -w tls_capture.pcap port 443
```

然后把 pcap 文件拖进 Wireshark 里分析。Wireshark 对 TLS 协议的解析做得很好，展开每一条消息能直接看到：

- Client Hello 里的 Cipher Suites 列表（一长串套件 ID）
- Server Hello 里选中的套件
- Certificate 里的证书链（从叶子证书到根 CA）
- Server Key Exchange 里的 ECDHE 参数（公钥、曲线类型等）

有意思的一点是 TLS 1.3 有个"中间盒兼容模式"（middlebox compatibility mode）。因为很多网络中间设备（防火墙、代理）只认 TLS 1.2 的格式，TLS 1.3 的 Client Hello 会故意伪装成 TLS 1.2 的 Record 格式，骗过中间设备。抓包的时候能明显看到一条 Change Cipher Spec 消息出现在不该出现的位置，那就是兼容模式在起作用。

## 几个实际遇到的问题

**1. 证书链不完整**

第三方服务器返回的证书链缺少中间 CA 证书，导致客户端本地没有对应的中间 CA 根证书去验证。解决办法是让对方在服务器配置里补全证书链，或者在客户端信任该中间 CA。

**2. 协议版本降级**

对方服务器默认支持 TLS 1.0 和 1.1，但我们客户端最低只接受 TLS 1.2。协商时对方会尝试降级，如果两边都不让步就握手失败。最后我们协商后让对方把 TLS 1.0/1.1 关掉了。

**3. SNI 扩展缺失**

有次发现抓包里 Client Hello 没带 Server Name Indication 扩展，导致服务端不知道该返回哪个虚拟主机的证书，返回了默认证书，客户端验证不通过就断了。原因是用的 openssl 命令行工具没指定 -servername 参数。

## 一些心得

通过这次实践，我对密码学在工程中的角色有了更具体的认识。之前觉得加密就是调个库的事，现在知道了：

- 加密套件的选择直接影响兼容性和安全性，不能随便配
- 证书验证是个完整的链路，中间任何一个环节断了都会出问题
- TLS 握手过程中的每个字段都有其作用，抓包分析是最直观的学习方式
- TLS 1.3 相比 1.2 确实在握手效率和安全性上都有提升，值得在新系统中优先采用

密码学不是只存在于教科书里的理论，它就藏在我们每天用的 HTTPS 连接里。下次遇到网络连接问题，不妨抓个包看看 TLS 握手过程，说不定问题就出在加密套件协商这一步。
