## 为什么要做这个


每天要看的信息源太多，手动刷RSS效率低，而且经常刷到一半被别的事打断。后来想能不能搭一个自动化工作流：定时抓取RSS源，用大模型筛选和总结，最后推送给自己。

选了Coze平台来实现。原因很简单：拖拽搭建，不用写太多代码，而且插件生态比较丰富，RSS相关的插件现成就有。

<!-- more -->
实际搭下来踩了不少坑，主要集中在三个方面：Coze的Python环境限制、RSS抓取的稳定性、以及Plugin部署的各种问题。下面把整个过程和解决方案记录一下。

## 工作流整体架构

整个工作流分四个核心环节：

```
RSS源抓取 → 文章正文采集 → LLM筛选与总结 → 结果输出
```

对应Coze里的节点：

1. **开始节点**：接收用户输入的RSS URL列表和兴趣关键词
2. **RSS读取插件节点**：批量抓取每个RSS源的文章列表
3. **代码节点**：解析RSS XML，提取文章链接和基本信息
4. **正文采集节点**：通过Plugin访问每篇文章的正文内容
5. **LLM节点**：基于兴趣关键词筛选文章，并生成摘要
6. **结束节点**：输出筛选后的文章列表和摘要

下面逐个讲每个环节的实现细节和遇到问题经验。

## RSS源抓取节点

Coze插件商店里搜"RSS"能找到好几个现成的插件，比如 `rss_reader`。这个插件的用法比较简单：输入一个RSS的URL，返回一个包含文章列表的JSON。

在工作流里用起来是这样：

- 插件节点选择 `rss_reader`
- 参数值设置为"引用"，选择开始节点的RSS URL输入参数
- 输出参数是一个JSON对象，里面包含 `articles` 数组

一般一个RSS源会返回最近30篇文章。如果同时订阅了多个源，需要用**批处理（Batch Processing）**来遍历每个RSS URL。

### 批处理的配置

批处理是Coze工作流里处理数组数据的关键机制。配置步骤：

1. 在RSS读取节点外面套一个循环节点
2. 循环数组参数引用开始节点的RSS URL列表（类型需要是Array）
3. 循环体里放RSS读取插件节点
4. 循环结束后，所有RSS源的文章会合并成一个大的文章列表

这里有个细节：如果开始节点的输入参数是字符串类型（一行一个URL），需要先用一个**文本处理节点**把它转成数组。在Python代码节点里处理：

```python
def main(args):
    rss_text = args.params['rss_list']
    urls = [url.strip() for url in rss_text.strip().split('\n') if url.strip()]
    return {'rss_urls': urls}
```

## Plugin节点：正文采集

RSS插件只能拿到文章列表（标题、链接、发布时间），拿不到正文内容。所以需要一个额外的节点来访问每篇文章的页面，提取正文。

这就是整个工作流里最麻烦的部分。

### 问题一：Coze内置的网页访问能力有限

Coze有一些内置的网页解析插件，但不是所有网站都能正常抓取。有些网站有反爬机制，有些需要JavaScript渲染才能拿到正文。

我的方案是自己部署一个Plugin，封装一个HTTP请求+HTML解析的逻辑。

### 问题二：Plugin部署遇到Python库安装问题

Coze的Plugin开发支持Python，但环境限制很大：

- **大部分三方库无法直接导入**。Coze的Python运行环境基于Pyodide（WebAssembly），只预装了少数几个库（`httpx`、`numpy`等）
- 即使是 `requests` 这种常用库，也需要确认是否在预装列表里
- `requests_async` 虽然支持，但已经停止维护多年，本地调试很麻烦

我一开始想用 `beautifulsoup4` 来解析HTML，在Coze环境里直接报错——找不到这个模块。

### 解决方案：用JavaScript替代Python

查了官方文档和社区帖子后发现，**Coze工作流的代码节点同时支持JavaScript和Python，而且JavaScript的限制更少**。

最终我用JavaScript重写了正文采集的逻辑：

```typescript
async function main(args: Args): Promise<Output> {
    const url = args.url;
    
    // 用原生fetch请求文章页面
    const response = await fetch(url);
    const html = await response.text();
    
    // 简单的正则提取正文（不依赖DOM解析库）
    // 根据目标网站的HTML结构定制提取规则
    let content = '';
    
    // 提取 <article> 标签内的内容
    const articleMatch = html.match(/<article[^>]*>([\s\S]*?)<\/article>/i);
    if (articleMatch) {
        content = articleMatch[1]
            .replace(/<[^>]+>/g, '')  // 去除HTML标签
            .replace(/\s+/g, ' ')    // 合并空白字符
            .trim();
    }
    
    return { content: content.substring(0, 5000) };  // 限制长度
}
```

JavaScript在Coze环境里用原生 `fetch` 就能发HTTP请求，不需要额外的库，而且没有Pyodide的那些限制。

### 调试流程

Coze的工作流**不支持在线调试代码节点**，所以必须在本地先调试好再贴进去。

本地调试步骤：

1. 安装NodeJS和VS Code
2. 创建项目，初始化JavaScript配置
3. 在本地写好代码并测试
4. 确认能跑通后，贴入Coze的代码节点main方法里

```bash
npm init -y
npm install typescript -g
npx tsc --init
# 编写代码后
npx tsc
node dist/index.js
```

本地跑通的代码贴到Coze里基本都能直接运行，这点比较靠谱。

## LLM节点：筛选与总结

拿到正文内容后，用LLM节点来做两件事：基于兴趣关键词筛选文章，以及生成摘要。

### 参数配置

LLM节点的配置要点：

- **模型选择**：Coze专业版一般用豆包系列模型，建议选 `pro-32k`，模式设为"更精准"
- **输入参数**：引用上游节点的文章列表（标题+正文+链接）和开始节点的兴趣关键词
- **输出格式**：要求LLM输出结构化的JSON

### 提示词设计

经过几轮调试，提示词的关键部分如下：

```
你是一个信息筛选助手。根据用户提供的兴趣关键词，从以下文章列表中筛选出相关的文章。

要求：
1. 阅读每篇文章的标题和正文，判断是否与用户的兴趣关键词相关
2. 对于相关的文章，生成一段简短的摘要（不超过100字）
3. 输出格式为JSON数组，每个元素包含：
   - article_link: 文章链接
   - title: 文章标题
   - summary: 摘要内容
   - relevance_score: 相关性评分（0-100）
4. 按relevance_score降序排列
5. 只输出JSON，不要输出其他内容
```

提示词设计有几个经验：

- **明确输出格式**。LLM节点的输出会被下游节点引用，如果格式不对会导致整个工作流报错。用JSON格式比较稳定，比Markdown格式更容易解析。
- **给具体的限制条件**。比如"摘要不超过100字"、"只输出JSON"，这样能减少LLM输出多余内容的情况。
- **用变量引用用户输入**。在提示词里用 `{{user_input}}` 引用开始节点的参数，这样每次运行可以传入不同的关键词。

### 处理LLM输出

LLM节点输出的是文本（JSON字符串），下游如果需要数组格式，需要用代码节点做一次转换：

```typescript
async function main(args: Args): Promise<Output> {
    const llmOutput = args.llm_output;
    let articles = [];
    
    try {
        articles = JSON.parse(llmOutput);
    } catch (e) {
        // 尝试提取JSON部分
        const jsonMatch = llmOutput.match(/\[[\s\S]*\]/);
        if (jsonMatch) {
            articles = JSON.parse(jsonMatch[0]);
        }
    }
    
    return { articles: articles };
}
```

这一步很重要，因为LLM有时候会在JSON前后加一些解释文字，直接 `JSON.parse` 会报错。加一个兜底的正则提取能提高稳定性。

## RSS抓取的稳定性问题

实际跑了一段时间后发现RSS抓取不太稳定，主要有几个原因：

### 问题一：部分RSS源返回格式不规范

有些网站的RSS feed格式不标准，比如XML声明缺失、标签嵌套错误等。Coze的RSS插件对这些问题的容错性不够好。

**解决方案**：在RSS读取节点后面加一个代码节点，做XML解析的容错处理。用正则表达式提取关键字段，而不是完全依赖XML解析器。

### 问题二：网络超时

有些RSS源响应比较慢，Coze的节点有超时限制，超过一定时间就会报错。

**解决方案**：在工作流里加一个错误处理分支。Coze支持条件判断节点，可以判断上游节点是否报错，报错的话跳过这个源继续处理下一个。

### 问题三：频率限制

Coze的插件调用有频率限制，批量抓取多个RSS源时容易触发。

**解决方案**：在批处理的循环体里加一个延迟节点（用代码节点的sleep），控制请求间隔。不过Coze的代码节点也有执行时间限制，sleep不能太长。实测500ms的间隔比较合适。

## 部署和运行

整个工作流搭好后，可以配置定时触发器让它自动运行：

1. 在Bot设置里选择"触发器"
2. 选择"定时触发"，设置运行时间（比如每天早上8点）
3. 配置输入参数的默认值（RSS URL列表、兴趣关键词）
4. 发布到飞书渠道（触发器目前只对飞书渠道生效）

也可以手动触发测试：在Coze的调试界面直接运行工作流，检查每个节点的输出是否正确。

## 总结

用Coze搭建信息采集工作流整体体验不错，拖拽搭建的效率比写代码快很多。但有几个点需要注意：

1. **Python环境限制大**。如果需要爬取网页或做复杂的数据处理，优先用JavaScript
2. **Plugin部署有坑**。三方库的安装在Coze环境里受限，最好先在本地确认好能用再部署
3. **RSS抓取要做容错**。不同网站的RSS格式差异大，网络也不稳定，需要加错误处理和重试逻辑
4. **LLM输出要格式化**。提示词里明确要求输出JSON，并加代码做格式转换，能避免很多下游节点的报错
5. **本地调试是必须的**。Coze不支持在线调试代码节点，所有代码都要在本地先跑通

目前这个工作流已经稳定运行了一段时间，每天自动抓取和总结几十篇文章，筛选出最相关的内容推送给我。省去了手动刷RSS的时间，信息获取效率提升了不少。
