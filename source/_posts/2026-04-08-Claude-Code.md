## 为什么开始用Claude Code

今年四月份开始用Claude Code。原因很直接——手上的项目越来越大，光是理解现有架构就要花不少时间，更别说做改动了。之前试过Cursor，也用过Copilot，但它们的定位更多是"代码补全"或者"在IDE里开个聊天窗"。Claude Code不一样，它跑在终端里，能直接操作文件系统、执行命令、读写代码。简单说，它不只是帮你写代码，它帮你"做项目"。

两个月下来，我对它的使用逐渐从"让它帮我改bug"变成了"让它帮我做架构决策"和"让它帮我管理整个重构流程"。这个转变，跟Claude Code的架构设计有很大关系。

## Claude Code的核心架构

Claude Code的架构思路跟很多AI编程工具不同。大多数工具走的是"多agent协作"路线——搞一堆小agent，一个负责写代码、一个负责测试、一个负责review。Claude Code反其道而行，核心就是一个单线程的主循环。

具体来说，它内部跑的是一个while循环：思考 → 调用工具 → 观察结果 → 再思考。循环不断转，直到模型判断任务完成、不再需要调用工具时终止。这个设计看起来简单，但实际用起来非常可控。因为你随时能看到它在做什么，也随时能打断它。

状态管理方面，Claude Code维持的是单一的消息历史记录，不搞复杂的对话分支。这意味着你不用担心它在某个分叉里跑偏了、回不来。用户可以在它执行过程中随时插入新指令——按Escape键打断，它会停下来等你重新指示。这个"随时转向"的能力在实际使用中非常有用，尤其是做架构调整的时候，方向变了直接说一声就行。

### 上下文构建方式

这是我觉得Claude Code最聪明的设计之一。它不像IDE插件那样预先索引整个代码库（那种方式在大项目里会很慢），而是像人一样，用grep、ls、find这些命令动态探索。第一次打开一个陌生项目时，它会先问你项目在哪，然后开始扫目录结构，读关键文件，逐步建立对项目的理解。

这种方式的好处是：它理解代码的过程是"渐进式"的，跟你自己看项目的方式很像。你告诉它"src/payment/下有个bug"，它会先ls那个目录，读相关文件，然后才开始改。不会一上来就读几百个文件然后说"我准备好了"——那样上下文窗口很快就满了。

### 工具体系

Claude Code内置了20多个工具，覆盖文件读写、代码编辑、shell命令执行、git操作等。此外支持MCP（Model Context Protocol）扩展外部工具，比如数据库查询、浏览器自动化、Slack消息发送。

但更关键的不是工具数量，而是权限管理。所有写操作和敏感命令都需要用户确认——你设好白名单之后，常用命令可以自动执行，其他的会弹出提示让你确认。这套机制让你敢于把比较大的任务交给它，因为你知道它不会偷偷改你不该改的东西。

### 上下文压缩

长时间使用时，对话历史会越来越长，逼近token上限。Claude Code有一套4级压缩策略：最便宜的是直接截断旧消息（Snip Compact），成本为零但丢信息；往上是选择性清理（Microcompact）、分层压缩（Context Collapse），最贵的是让AI总结历史（Auto-Compact）。

实际使用中，我大概每30-40条消息会跑一次`/compact`，手动触发一下。它会把之前的对话总结成摘要，腾出上下文空间。这个设计让长会话不会因为上下文溢出而崩溃。

## CC Switch：多模型切换的效率工具

Claude Code本身只支持Claude系列模型（Opus、Sonnet、Haiku），但在国内使用时经常遇到额度用完、网络不稳定等问题。CC Switch是一个开源工具，专门解决多模型切换的痛点。

它的核心功能很简单：在界面上配好多个模型供应商（Claude官方、智谱GLM、Kimi K2、DeepSeek等），点一下就能切换。不需要手动改配置文件，不需要重启终端。

### 本地代理机制

CC Switch的原理是在本机启动一个本地代理服务。你把Claude Code的API地址指向`127.0.0.1:15721`（CC Switch的本地端口），所有请求先经过CC Switch转发，再到达你选的模型供应商。切换模型时，CC Switch直接改转发目标，Claude Code这边完全无感。

这种方式的好处是：
- 配置一次就行，之后都是点一下切换
- 支持多设备同步（通过坚果云等云盘同步配置目录）
- WSL环境也能用，Windows和Linux共享同一套配置

### 实际使用场景

我现在的使用模式是：日常开发用Sonnet（便宜、速度快），做架构设计时切到Opus（推理能力更强），处理简单任务时用国产模型（额度更充裕）。一天切个五六次是常态，如果手动改配置的话确实很烦，CC Switch基本上5秒搞定。

有一个细节值得一提：Claude Code支持用`/model`命令在会话内切换模型。但这个切换有时候会在Opus额度用完后自动降级到Sonnet，而且不通知你。如果你正在做需要深度推理的任务，降级后质量会明显下降。CC Switch的好处是可以精确控制用哪个模型，不会被自动降级。

## 架构设计能力：Plan Mode

这是我最看重的功能。Claude Code有一个Plan Mode（规划模式），按两次Shift+Tab进入。进入后，Claude不会直接写代码，而是先分析你的需求，输出一份方案让你审核。方案里会列出它打算改哪些文件、用什么策略、可能的风险点。

这个流程跟人类架构师的工作方式很像：先出方案，确认了再动手。Anthropic的开发者关系负责人说过他90%的时间都在用Plan Mode。

### 实际操作

举个例子。我有一次需要把一个单体服务拆分成微服务。用Plan Mode，我告诉Claude："把src/下的服务拆成三个独立模块：auth、payment、notification。每个模块独立部署，共享数据库但各自有自己的表前缀。"

Claude花了大概两分钟分析项目结构，然后输出了一份方案：先识别所有数据库查询，按模块分组；然后创建三个独立的服务目录；最后修改路由和依赖注入。方案里还标出了几个需要手动处理的地方（比如共享的中间件）。

确认方案后，Claude开始执行。整个过程大概跑了40分钟，期间我按了几次Escape做了调整（比如它一开始想把某个utils模块直接复制三份，我让它改成共享库）。最终产出的代码结构清晰，测试全部通过。

### 跟其他工具的对比

Cursor和Copilot也能做类似的事情，但它们的"规划"更多是在对话框里说说，不会真的帮你拆解任务、逐步执行。Claude Code的不同在于：它是真的在操作你的代码库——读文件、改文件、跑测试、commit。整个过程是"可执行的方案"，不是"建议你这样做"。

## 项目管理能力：CLAUDE.md和SubAgent

### CLAUDE.md：项目的记忆卡

每个项目根目录放一个`CLAUDE.md`文件，里面写项目的基本信息：目录结构、构建命令、代码规范、常用命令等。Claude Code启动时会自动读取这个文件，相当于给它一份"项目手册"。

用`/init`命令可以让Claude自动扫描项目并生成`CLAUDE.md`模板。之后你可以根据实际情况修改。我一般会写这些内容：

- 项目用的技术栈和版本
- 目录结构说明
- 测试命令（`npm test`、`pytest`等）
- 代码规范（ESLint配置、Python的ruff等）
- 常见的坑和注意事项

这个文件让Claude Code不用每次重新"认识"你的项目。你跟它说"修一下auth模块的bug"，它立刻知道auth模块在`src/auth/`下，测试用`npm test -- --grep auth`。

### SubAgent：分工协作

SubAgent是Claude Code在7月推出的特性。简单说就是让Claude Code创建独立的子agent，每个子agent有自己的指令、工具权限和上下文。

一个典型的用法是：让一个SubAgent专门做代码审查（只读权限，不能改代码），另一个SubAgent专门写测试。这样主线程可以继续做其他事情，不用等它们完成。

我一般会配置3-4个SubAgent：
- code-reviewer：代码审查，只读
- test-writer：写单元测试
- doc-writer：更新文档

配置方式是在`.claude/agents.json`里写，每个agent指定描述、模型、指令、权限。用的时候在对话里说"用code-reviewer审查刚才的改动"就行。

### Background Tasks

用`&`后缀可以把任务放到后台执行。比如`refactor API routes &`会启动一个后台agent重构路由，你可以同时做其他事情。多个后台agent可以在隔离的Git worktree里并行工作，互不干扰。

## 两个月的整体感受

Claude Code不是那种"装上就飞"的工具。它需要你花时间理解它的工作方式，配置好CLAUDE.md，想清楚什么任务交给它、什么任务自己做。

两个月下来，我的使用习惯是这样的：

1. **架构设计**：用Plan Mode。先描述需求，看Claude的方案，确认后再执行。这是它最值钱的能力。
2. **日常开发**：直接用自然语言描述需求，让它写代码。简单的功能实现非常快，复杂的逻辑需要多轮对话。
3. **重构**：先用CLAUDE.md描述清楚项目结构，然后给Claude一个明确的重构目标。它会自己探索代码、制定计划、逐步执行。
4. **测试**：用SubAgent写测试。让一个专门的agent负责测试，不会干扰主线程的开发。
5. **模型选择**：日常用Sonnet，需要深度推理时切Opus。CC Switch让切换成本几乎为零。

最大的收获是：Claude Code帮我建立了"先规划、后执行"的工作习惯。以前我经常是一边写代码一边想架构，写到一半发现方向错了要重来。现在我会先用Plan Mode让Claude出方案，确认了再动手。这个习惯节省的时间远比Claude Code帮我写代码省的多。
