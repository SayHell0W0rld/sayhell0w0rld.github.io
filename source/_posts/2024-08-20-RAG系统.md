最近搭了一个本地跑的 RAG 系统，技术栈是 LangChain + ChromaDB。记录一下搭建过程、搜索原理和调参经验，给想自己动手搭的人一个参考。

## 先说背景

大语言模型（LLM）的训练数据是有截止日期的，也不可能包含你的私有数据。比如我有一批简历，想让模型根据简历内容回答问题——"这个候选人会 Python 吗？""哪几个人有 3 年以上经验？"——直接问 LLM 肯定不行，它没见过这些文件。

RAG（Retrieval-Augmented Generation，检索增强生成）的做法是：先把文档切块、编码成向量存进向量数据库，用户提问时先检索相关内容，再把这些内容拼进 prompt 让 LLM 回答。这样 LLM 就能基于你的私有数据来生成答案了。

整个流程分两个阶段：

1. **离线索引阶段**：加载文档 → 切分成 chunk → 用 embedding 模型编码成向量 → 存入 ChromaDB
2. **在线查询阶段**：用户提问 → query 也编码成向量 → 向量搜索找到最相关的 chunk → 拼成 prompt → 调用 LLM 生成回答

## 环境搭建

我用的环境：

```bash
pip install langchain langchain-community chromadb sentence-transformers ollama
```

- **LangChain**：负责串联整个 RAG pipeline
- **ChromaDB**：本地向量数据库，纯 Python 实现，不需要额外部署服务
- **sentence-transformers**：本地 embedding 模型，不需要调 API
- **Ollama**：本地跑 LLM（比如 llama3.1:8b），也可以换成任何你习惯的 LLM

如果你用 OpenAI 的 embedding 和 LLM，把 `sentence-transformers` 换成 `openai` 就行，ChromaDB 那边 LangChain 有现成的集成。

## 数据加载与切分

先加载文档。LangChain 支持 PDF、TXT、Markdown 等格式，简历一般是 PDF：

```python
from langchain_community.document_loaders import PyPDFLoader

# 加载单个简历
loader = PyPDFLoader("resume_001.pdf")
docs = loader.load()

# 批量加载整个目录
from langchain_community.document_loaders import DirectoryLoader

loader = DirectoryLoader("./resumes/", glob="*.pdf", loader_cls=PyPDFLoader)
docs = loader.load()
print(f"共加载 {len(docs)} 页文档")
```

然后切分。这是整个 RAG 系统里最关键的一步——切得好不好，直接决定检索质量。

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,       # 每个 chunk 最大 800 字符
    chunk_overlap=200,    # 相邻 chunk 重叠 200 字符
    separators=["\n\n", "\n", "。", "；", " ", ""]
)
chunks = text_splitter.split_documents(docs)
print(f"共生成 {len(chunks)} 个 chunk")
```

**为什么要切分？** 两个原因：

1. Embedding 模型对输入长度有限制，太长会截断
2. 检索粒度太粗的话，一大段无关内容会稀释相关信息，LLM 拿到的上下文质量变差

**chunk_size 和 chunk_overlap 怎么选？** 我后面会详细说。

## Embedding 与向量存储

切分完成后，用 embedding 模型把每个 chunk 编码成向量，存入 ChromaDB：

```python
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import OllamaEmbeddings

# 用 Ollama 跑本地 embedding
embedding = OllamaEmbeddings(model="nomic-embed-text")

# 存入 ChromaDB（首次创建，自动持久化）
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embedding,
    persist_directory="./chroma_db"
)
```

ChromaDB 会把向量和原始文本一起存在本地 `./chroma_db` 目录。下次启动不用重新 embedding，直接加载：

```python
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embedding
)
```

这个持久化机制挺方便，上万份简历编码一次就够了，后面查询都是毫秒级。

## 检索：向量搜索是怎么工作的

RAG 的核心在"Retrieval"——怎么从一堆 chunk 里找到和问题最相关的那几条。

### 向量搜索（Dense Search）

embedding 模型把文本编码成一个低维稠密向量（比如 nomic-embed-text 输出 768 维）。向量之间的距离（通常是 cosine similarity）代表语义相似度。

```python
# 手动测试语义搜索
results = vectorstore.similarity_search("会 Python 的候选人", k=3)
for doc in results:
    print(doc.page_content[:100])
    print("---")
```

为什么叫"语义搜索"？因为它的比较不是基于关键词匹配，而是基于含义。比如你搜"编程语言经验"，即使用的是"Python"这个词出现频率不高的 chunk，只要那个 chunk 在语义上和"编程语言"相关，也能被检索到。

### Sparse Search（关键词搜索）

传统的关键词搜索（如 BM25）是另一种方式。它基于词频统计，精确匹配关键词。

两者对比：

| 维度 | Dense Search（向量搜索） | Sparse Search（关键词搜索） |
|------|------------------------|--------------------------|
| 匹配方式 | 语义相似度（cosine） | 关键词词频（TF-IDF） |
| 处理同义词 | 好，"编程语言"能匹配"Python" | 差，必须精确命中 |
| 处理精确术语 | 一般，可能漏掉专有名词 | 好，"etcd server.key"精确匹配 |
| 计算速度 | 向量计算，稍慢 | 倒排索引，很快 |
| 适用场景 | 自然语言问题、模糊查询 | 精确技术术语、编号搜索 |

在我的简历筛选场景中，大部分查询是自然语言（"有机器学习经验的人"），用 Dense Search 效果更好。但有些查询是精确术语（"要求 CKA 认证"），Dense Search 可能漏掉。实际使用中可以两种结合。

### LangChain 中的检索配置

```python
retriever = vectorstore.as_retriever(
    search_type="similarity",   # 默认 cosine similarity
    search_kwargs={"k": 5}      # 返回 top-5 结果
)
```

LangChain 支持几种 search_type：

- **similarity**：默认，返回最相似的 k 个 chunk
- **mmr**（Maximal Marginal Relevance）：在相关性基础上增加多样性，避免返回重复内容
- **similarity_score_threshold**：只返回超过阈值的结果

对于简历筛选这种场景，我用 `similarity` + 合理的 k 值就够了。

## 生成回答

检索到相关 chunk 后，把它们拼进 prompt 让 LLM 生成回答：

```python
from langchain_community.chat_models import ChatOllama
from langchain.chains import RetrievalQA

llm = ChatOllama(model="llama3.1:8b", temperature=0)

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",       # 把所有检索结果拼接成一个 prompt
    retriever=retriever,
    return_source_documents=True   # 同时返回引用的文档来源
)

# 提问
result = qa_chain.invoke({"query": "这些候选人中有谁会 Rust 吗？"})
print(result["result"])
```

`chain_type="stuff"` 是最简单的策略：把检索到的 chunk 直接拼接在一起塞进 prompt。如果检索结果太多，可能会超过 LLM 的上下文长度限制，这时候可以换用 `map_reduce` 或 `refine` 等策略。

我实际测试下来，k=3~5 比较合适。太少可能漏掉信息，太多会引入噪声。

## 调参技巧

搭起来容易，调好需要耐心。我把遇到的问题整理一下：

### 1. chunk_size

这是影响最大的参数。

- **太小**（<300 字符）：chunk 太碎，丢失上下文，LLM 读到的每条信息都不完整
- **太大**（>2000 字符）：chunk 太粗，相关信息和不相关信息混在一起，检索精度下降
- **经验值**：简历场景下 600~1000 字符比较合适。一般文档 800~1200 是个不错的起点

### 2. chunk_overlap

重叠区域让相邻 chunk 的语义连续，避免信息在边界处断裂。

- **太小**（<50）：边界处信息丢失严重
- **太大**（>chunk_size 的 30%）：冗余过多，浪费存储和计算
- **经验值**：chunk_size 的 15%~25%，比如 chunk_size=800 时 overlap=150~200

### 3. k 值（检索数量）

- **k=3**：适合简单查询，信息量少
- **k=5~8**：适合复杂查询，需要多条信息综合回答
- **k 太大**：引入噪声，LLM 回答反而变差
- **我自己的做法**：先设 k=5，根据回答质量微调

### 4. Embedding 模型选择

不同 embedding 模型对检索质量影响很大：

- **nomic-embed-text**：开源，本地跑，质量不错，推荐入门
- **bge-large-zh-v1.5**：中文场景效果更好
- **text-embedding-3-small**（OpenAI）：质量高，但需要 API key 和费用
- **sentence-transformers/all-MiniLM-L6-v2**：轻量，适合英文场景

简历筛选涉及中文，我用的是 bge-large-zh-v1.5，效果比 nomic-embed-text 好一些。

### 5. LLM 的 temperature

temperature=0 让回答确定性更强，适合需要精确引用的场景。temperature 越高回答越"灵活"，但可能编造不存在的信息。RAG 场景一般设 0 或 0.1。

## 完整代码示例

把上面的步骤串起来，一个最小可运行的 RAG 系统大概是这样：

```python
import os
from langchain_community.document_loaders import DirectoryLoader, PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import OllamaEmbeddings
from langchain_community.chat_models import ChatOllama
from langchain.chains import RetrievalQA

# 1. 加载文档
loader = DirectoryLoader("./resumes/", glob="*.pdf", loader_cls=PyPDFLoader)
docs = loader.load()
print(f"加载了 {len(docs)} 页")

# 2. 切分
splitter = RecursiveCharacterTextSplitter(
    chunk_size=800, chunk_overlap=200,
    separators=["\n\n", "\n", "。", "；", " ", ""]
)
chunks = splitter.split_documents(docs)
print(f"生成 {len(chunks)} 个 chunk")

# 3. Embedding + 向量存储
embedding = OllamaEmbeddings(model="nomic-embed-text")
vectorstore = Chroma.from_documents(
    chunks, embedding, persist_directory="./chroma_db"
)

# 4. 构建 QA 链
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
llm = ChatOllama(model="llama3.1:8b", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm, chain_type="stuff",
    retriever=retriever, return_source_documents=True
)

# 5. 提问
while True:
    query = input("\n请输入问题（输入 q 退出）：")
    if query.strip().lower() == 'q':
        break
    result = qa_chain.invoke({"query": query})
    print(f"\n回答：{result['result']}")
    print("\n引用来源：")
    for doc in result["source_documents"]:
        print(f"  - {doc.metadata.get('source', 'unknown')}: {doc.page_content[:80]}...")
```

## 写在最后

搭下来的感受：RAG 系统的核心不在 LLM，而在数据处理和检索质量。embedding 模型选对、chunk 切好、k 值调对，回答质量就能到一个不错的水平。LLM 本身反而是最容易替换的部分。

如果后续想继续优化，几个方向可以考虑：

- **混合检索**：Dense Search + Sparse Search 结合，覆盖语义和精确匹配
- **Reranking**：检索后再用 reranker 模型对结果重新排序
- **Query 改写**：用户原始 query 不太适合检索时，先用 LLM 改写一下
- **Fine-tune embedding**：用你的领域数据微调 embedding 模型，提升特定场景的检索效果

这些属于进阶优化了，先把基础版跑通，再按需加。
